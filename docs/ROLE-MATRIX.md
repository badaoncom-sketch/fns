## 0. 개요

본 문서는 [PRD.md](PRD.md) §5.6.4(관리자 권한 체계)에서 정의한 7개 역할이 ERP 각 모듈(메뉴)에서 수행할 수 있는 액션을 매트릭스로 정리한 것이다. **본 문서는 새로운 권한 규칙을 만들지 않는다** — PRD.md/DECISIONS.md/ARCHITECTURE.md에 이미 확정된 제약은 그대로 인용하고, 확정되지 않은 셀은 "미확정"으로 표기하며 관련 Open Decision 번호를 인용한다.

### 0.1 역할 7종 (PRD.md §5.6.4 인용)

| 역할 | 스코프 | 설명 (PRD.md §5.6.4 원문) |
|---|---|---|
| **SuperAdmin** | 전체 국가 | 모든 기능 접근. 조직 이동(§5.16) 수행 가능 |
| **Compliance Admin** | 전체 국가 (또는 국가별, 미확정) | 법무/컴플라이언스 전담 관리자. 조직 이동(§5.16) 수행 가능 — 역할 신설 (D-020) |
| **지정 조직관리 관리자** | 별도 지정(미확정 — 국가별/전체) | 조직 구조 관리 전담으로 개별 지정된 관리자만 부여. 조직 이동(§5.16) 수행 가능 — 역할 신설 (D-020) |
| **CountryAdmin** | 단일 국가 | 본인 국가의 회원/정산/보고서만 접근. 조직 이동 권한 없음(일반 관리자로 분류, D-020) |
| **CenterAdmin** | 단일 센터 (§5.9, 도입 자체 미확정) | 본인 센터의 회원/매출만 접근. 조직 이동 권한 없음 |
| **기능별 관리자** (정산담당/법무담당/고객지원/물류담당 등) | 기능 단위, 국가/센터와 결합 가능 | 역할 목록·권한 매핑 미확정(O-042). 조직 이동 권한 없음(법무담당↔Compliance Admin 동일 여부도 미확정 — O-066) |
| **Partner** (회원) | 본인 데이터 + 본인 하위 추천조직 조회 | 회원 본인 화면(마이오피스/내 조직/회원몰/쇼핑몰). 조직 이동 등 관리자 전용 기능에는 어떤 형태로도 접근하지 않음(D-020) |

> SuperAdmin을 제외한 모든 관리자는 본인의 국가 스코프를 벗어난 데이터에 접근할 수 없다 (PRD.md §5.6.4, [ARCHITECTURE.md](ARCHITECTURE.md) §4). CenterAdmin은 본인 센터로 추가 제한된다.

조회 / 등록 / 수정 / 삭제 / 승인 / 반려 / 출력 / 다운로드 / 엑셀 / 일괄처리 / 권한관리

### 0.3 표기 규칙

| 표기 | 의미 |
|---|---|
| O | 확정 — 해당 역할이 해당 액션을 수행할 수 있다 |
| X | 확정 — 해당 역할이 해당 액션을 수행할 수 없다 |
| △ | 조건부 — 사유를 셀 옆 각주로 명시 |
| 미확정 | 확정된 근거 문서가 없음 — 관련 Open Decision 번호 병기 |
| (해당없음) | 그 모듈/액션 조합 자체가 의미 없음(예: Partner에게 "권한관리" 액션) — 별도 표기, 미확정과 구분 |

> 모든 셀을 임의로 채우지 않는다. 이미 확정된 제약(조직 이동 3역할 한정, CountryAdmin 국가 스코프 한정 등)은 그대로 반영하고, 그 외 다수 셀은 PRD.md/DECISIONS.md에 아직 역할별 권한 매핑이 없으므로 **미확정**으로 둔다 — 이는 본 매트릭스의 결함이 아니라 설계 문서의 현재 상태를 정확히 반영한 것이다(§33 참조).

### 0.4 공통 전제 (모든 표에 적용)

- 기능별 관리자의 정확한 종류·권한 매핑은 전체 미확정(O-042) — 아래 표의 "기능별 관리자" 열은 해당 모듈과 직접 관련된 기능(예: 정산관리↔정산담당)이 있을 경우에만 △로 표기하고, 그 외는 모두 미확정으로 둔다.
- CenterAdmin 관련 셀은 센터 구조 도입 자체가 미확정(PRD.md §5.9)이므로, 도입 시에만 유효하다는 전제를 깐다 — 표기는 도입을 가정한 "예정값"이며 별도 각주로 구분한다.
- Partner(회원)는 관리자 메뉴에 원천적으로 접근하지 않는다 — 회원 전용 화면(마이오피스/내 조직/회원몰/쇼핑몰/CS)에서의 본인 데이터에 대한 액션만 표기한다.

1. 회원 삭제(물리삭제)는 §5.4 민감 변경(탈퇴/강제탈퇴) 절차를 통해서만 가능하며 즉시 삭제 기능이 아니다 — "삭제" 액션은 사실상 강제탈퇴 승인 흐름으로 대체된다. SuperAdmin만 강제탈퇴까지 포함한 전체 흐름에 접근 가능하다는 전제이며, 정확한 삭제 권한 분배는 미확정(O-027 강제탈퇴 사유코드/이의제기 체계 미확정과 연동).
2. Compliance Admin은 법무/컴플라이언스 전담 관리자로(PRD.md §5.6.4) 회원 명의변경/강제탈퇴 등 민감 변경 승인에 관여할 개연성이 높으나, §5.4 민감 변경 승인자가 역할별로 어떻게 배정되는지는 PRD.md에 명시되어 있지 않다 — 미확정(O-042)으로 보는 것이 정확하나, 컴플라이언스 성격상 조회/승인/반려 가능성을 △로 표기.

7. Partner의 회원정보 수정은 "이름 외 연락처/주소 등 일반 정보"(PRD.md §5.3)에 한정되며 변경 이력이 보존된다 — 명의변경/탈퇴 등 민감 변경은 신청만 가능하고 즉시 반영(수정)되지 않는다(승인 필요).
8. **D-073 보강 — Customer Timeline(회원 상세의 신규 탭, PRD.md §5.64)은 신규 권한 표면이 아니다.** 가입/주문/결제/배송/문의/반품/패키지구매/정산/Lifestyle/로그인/알림을 한 화면에 모아 보여주는 통합 타임라인은 기존 "회원 상세" 화면(위 "조회" 열이 다루는 화면)에 추가되는 읽기 전용(read-only) 탭일 뿐이며, 여러 기존 테이블(§3 주문관리/§4 배송관리/§5 정산관리/§10 CRM/§22 알림로그 등)에 대한 federated 조회이므로 그 데이터 자체도 각 원본 모듈의 조회 권한 범위를 벗어나지 않는다. 따라서 Customer Timeline 열람 권한은 위 표의 "조회" 셀과 완전히 동일하며 별도 행·별도 셀을 신설하지 않는다 — 즉 회원 상세를 조회할 수 있는 역할(SuperAdmin/Compliance Admin△²/CountryAdmin△⁴ 등)은 동일한 범위로 Customer Timeline도 볼 수 있고, 조회할 수 없는 역할은 마찬가지로 볼 수 없다.

4. **상품별 SEO(`product_seo`)는 §2 상품관리 권한과 동일선상에서 다뤄야 정합성이 유지된다** — 즉 상품을 등록/수정할 수 있는 역할이라면 그 상품의 SEO 필드도 함께 다룰 개연성이 높다(상품 등록/수정 화면의 일부 탭일 가능성). 반면 `tenant_share_images`(테넌트 단위 공유 이미지)와 `tenant_settings`의 사이트 전역 SEO 컬럼(DATABASE.md §3.55)은 테넌트 최상위 설정이므로 SuperAdmin 중심으로 보는 것이 §19(System Settings)·§20(Multi-Tenant 설정)과 더 정합적이다. `content_seo_metadata`(페이지별 SEO)는 대상이 CMS 페이지/Marketing Program/FAQ 등 다양해 §9(CMS)·§11(Marketing Program) 각각의 기존 권한을 따를 개연성이 있다. 이처럼 SEO 관리는 단일 화면이 아니라 기존 여러 모듈에 걸친 횡단 기능이라, "SEO 관리자"라는 이름의 전담 역할이 별도로 신설될지, 아니면 각 모듈 담당자가 자기 영역의 SEO 필드만 함께 관리하게 될지가 불명확하다 — 만약 전담 역할이 도입된다면 PRD.md §5.6.4가 정의한 기존 7역할 프레임 안에서는 "기능별 관리자"의 한 하위유형(SEO 관리자 후보)으로 분류되어야 하며, 이는 O-042(기능별 관리자 역할 목록·권한 매핑 미확정)의 적용 범위에 포함된다 — 신규 최상위 역할로 만들지 않았다. §33 항목 8 참조.
5. Partner는 관리자용 SEO 입력 화면에 접근하지 않는다 — Partner가 보는 것은 SEO의 결과물(검색엔진 노출, 공유 시 미리보기 등)뿐이며 이는 PRD.md §5.46.4 SEO Preview가 순수 프론트엔드 기능임과 일관된다.

7. Partner는 운영자 대시보드/관리자 업무 Queue에 접근하지 않는다 — 관리자 전용 화면이며 회원의 자기 데이터 조회(마이오피스 등)와는 무관하다.
8. **D-073 보강 — §5.59(운영자 대시보드) 위젯 2종(SEO 오류/Feed 오류, `product_seo`/`scheduled_job_run_logs` 파생 쿼리) 및 Abandoned Cart 위젯(§5.62, `cart_items`/`carts`/`notification_logs` 파생 쿼리)은 모두 위 각주1·2가 이미 정의한 "조회" 범위 안에 있다.** SEO 오류/Feed 오류 위젯은 §24(SEO/공유이미지 관리)·§25(Digital Marketing 연동/Feed 관리)가 다루는 기존 테이블의 집계 조회일 뿐 신규 데이터 모델이 아니며, Abandoned Cart 위젯도 §22(Notification Center) 각주8과 동일한 기존 테이블 조회다. 따라서 세 위젯 모두 신규 행·신규 열을 추가하지 않으며, 위젯 자체의 등록/수정/삭제는 각주2와 동일하게 Dashboard Builder(§16)의 기존 권한 구조를 그대로 따른다.
9. **D-073 보강 — Quick Action(§5.61의 구체화: 주문조회/회원조회/상품등록/패키지등록/송장등록/환불승인/공지등록)은 신규 권한 표면을 만들지 않는다.** 각 액션은 단순 라우팅 단축키일 뿐이며 실행 시 이동하는 화면의 기존 권한 셀(§3 주문관리 "조회", §1 회원관리 "조회", §2 상품관리 "등록", §4 배송관리 "등록", §3 주문관리 "승인"(각주5 환불 승인), §9 CMS "등록")을 그대로 적용받는다 — 기존 승인 워크플로우를 우회하지 않는다(PRD.md §5.61 명시). 별도 "Quick Action" 권한 셀이나 신규 섹션을 두지 않는다.

## 27. Board Engine (게시판 엔진)

PRD.md §5.67(Dynamic Board Engine, D-074 — 공지사항/보도자료/갤러리/FAQ/자료실/이벤트/홍보영상/교육자료/제품자료/인증자료/사용후기/회사소개/CSR/채용/IR 등을 관리자가 코드 배포 없이 직접 생성하는 범용 게시판 엔진)를 포함. 관련 테이블: `boards`/`board_categories`/`board_posts`/`board_post_comments`/`board_post_likes`(DATABASE.md §3.58, 신규 5종). **본 모듈은 기존 CMS(§9 — `cms_pages`/FAQ/팝업/배너)를 변경하지 않으며, 그 옆에 신설되는 완전히 별개의 병렬 구조다** — `boards`/`board_posts`는 `cms_pages`와 다른 테이블이므로 권한도 §9와 별도로 평가한다(각주 a 참조). PRD.md §5.67.7은 **"신규 권한 테이블을 만들지 않는다 — 기존 7역할×11액션 체계를 게시판 모듈에 그대로 적용한다"**고 명시하므로, 본 절은 새로운 권한 규칙을 만드는 것이 아니라 기존 체계를 Board Engine이라는 신규 데이터 엔터티(게시판/게시글)에 매핑하는 것이다.

아래 표는 **게시판(`boards`) 레벨 액션**(게시판 생성/수정/비활성·활성/메뉴노출 설정 등 §5.67.2 관리자 설정 항목 전체)을 다룬다. 게시글(`board_posts`)·댓글(`board_post_comments`)·좋아요(`board_post_likes`) 레벨 액션은 표 아래 "게시글 레벨 액션"(P1~P6)에서 별도로 다룬다(같은 7역할×11액션 틀을 게시글 단위로 다시 적용한 것일 뿐 별도 표를 만들지 않는다).

1. 게시판(`boards`) 자체의 삭제는 그 하위 `board_posts`를 다수 보유할 수 있어 소프트 삭제(`is_active=false`로 비활성화) 우선 원칙(DATABASE.md §4 원칙 3)을 따를 개연성이 높다 — 물리 삭제 허용 여부와 정확한 정책은 미확정이나 SuperAdmin은 PRD.md §5.6.4 "모든 기능 접근" 원칙에 따라 O로 표기.
2. 게시판 정의(`boards`) 자체에는 승인 워크플로우가 없다(PRD.md §5.67.2에 검토/승인 단계 언급 없음, §9 CMS 각주1과 동일한 직접 편집 구조로 추정) — "승인"/"반려"는 게시판 레벨에는 해당없음. 게시글(`board_posts`) 레벨의 승인은 아래 P3에서 별도로 다룬다.
3. CountryAdmin은 §9(CMS) 각주3과 동일한 논리로, `boards.country_codes`로 본인 국가에 노출되는 게시판에 대한 조회 개연성이 있으나 등록/수정 권한 여부는 미확정.
4. Partner는 게시판 관리자 설정 화면(생성/수정/메뉴노출 등)에는 어떤 형태로도 접근하지 않는다(해당없음) — 표기된 △는 게시판이 노출하는 **게시글 목록**(예: 보도자료/자료실/FAQ)의 조회를 의미하며, 비회원도 조회 가능한지(공개형) 또는 회원 전용인지는 게시판마다 다르다(각주 a 참조). 출력/다운로드도 동일하게 게시판별 공개범위에 종속된다.

**게시글(`board_posts`) 레벨 액션** — 위 7역할×11액션 틀을 게시글 단위로 다시 적용한다(별도 표 없음, 게시판 레벨 표의 역할별 등록/수정/조회 권한과 기본적으로 동일선상이되 아래 세부 사항이 추가된다):

P1. **게시글 등록/수정**: 게시판 등록/수정 권한을 가진 역할(위 표 SuperAdmin O, 그 외는 미확정/△/X)이 그 게시판 산하 게시글도 등록/수정한다고 추정 — 별도 셀을 신설하지 않는다. **예약게시(`scheduled_publish_at`)는 별도 액션이 아니라 등록/수정 액션의 한 옵션이다**(PRD.md §5.67.7 명시) — §4(배송관리) 각주5·§22(Notification Center) 각주4와 동일하게 "기존 등록/수정 열에 이미 포함" 원칙을 적용한다.
P2. **게시글 삭제**: 게시판 삭제와 동일한 소프트 삭제 우선 원칙(위 표 각주1)을 따른다.
P3. **게시글 승인**: `feature_flags.승인후게시=true`로 설정된 게시판에서만 의미를 갖는다(PRD.md §5.67.7) — 이 경우 Workflow Engine(§12)을 재사용하며 `workflow_instances.subject_type='BOARD_POST_APPROVAL'`(DATABASE.md §3.58, 기존 자유 카탈로그 값 추가, `PRODUCT_APPROVAL`과 동일 패턴)로 처리된다. 즉 **신규 승인구조가 아니다** — 누가 승인자가 되는지는 §12 각주2와 동일하게 "워크플로우 설정에 종속"(△)이며 본 매트릭스가 고정하지 않는다. `feature_flags.승인후게시=false`인 게시판(대부분의 공지/자료실형 예상)은 승인 액션 자체가 해당없음.
P4. **첨부파일/다운로드**: File Manager(§14)의 기존 권한 패턴을 재사용한다(`files.related_entity_type='board_posts'`, DATABASE.md §3.58) — 별도 권한 구조를 신설하지 않으며, §14 표의 역할별 매핑(대부분 미확정, SuperAdmin O)을 그대로 따른다.
P5. **댓글/답글**: `boards.feature_flags.댓글=true`인 게시판에서만 사용(`board_post_comments`, `parent_comment_id` 자기참조로 답글 표현). **회원(Partner)은 본인이 작성한 댓글/답글만 수정·삭제할 수 있다** — 위 표의 Partner "수정"/"삭제" 셀(게시판 레벨은 X)과 별개로, 댓글 레벨에서는 본인 댓글에 한해 △(등록 O, 본인 댓글 수정·삭제 O, 타인 댓글 수정·삭제 X)로 본다. 관리자(게시판 등록/수정/삭제 권한을 가진 역할)는 모든 댓글에 대해 조회/수정/삭제(모더레이션) 가능 — 댓글 모더레이션 정책(예: 신고 임계치 자동 비노출 여부)은 PRD.md §5.67에 명시가 없어 별도 미확정 항목으로 §33에 추가했다(§33 항목 11 참조).
P6. **좋아요**: `boards.feature_flags.좋아요=true`인 게시판에서만 사용(`board_post_likes`, 회원당 게시글당 1회 unique). Partner는 본인 좋아요만 등록(누르기)·삭제(취소) 가능 — 타 회원의 좋아요를 대신 등록/취소할 수 없다. 관리자는 집계 조회만 가능하고(좋아요 수는 `board_posts.like_count_cache`) 타 회원의 좋아요 자체를 직접 등록/취소하는 액션은 일반적으로 의미가 없다(해당없음).

> a. **공개범위(국가/언어/노출 위치)는 역할 매트릭스가 아니라 게시판 설정값이다.** `boards.country_codes`/`language_codes`/`menu_exposure`/`shop_exposure`/`shop_member_exposure`/`my_office_exposure`/`main_exposure`(DATABASE.md §3.58)는 게시판 생성 시 관리자가 지정하는 **컬럼값**이며, "어떤 게시판이 비회원에게도 보이는지"는 그 컬럼값에 의해 board마다 다르게 결정된다(예: 보도자료/자료실/IR은 비회원 공개형일 개연성이 높고, 일부 사용후기/CSR형은 회원 전용으로 운영될 수 있음) — 본 매트릭스는 그 실제 노출 결과를 표기하는 것이 아니라 **"관리자가 그 노출 설정을 할 수 있는지"(위 표의 등록/수정 권한)만** 다룬다. Partner 행의 게시글 조회 △ 역시 이 컬럼값에 따라 게시판별로 달라질 수 있다는 전제 위에 표기된 것이다.
> b. **§9(CMS)와 본 §27(Board Engine)은 완전히 별개의 권한 표다** — 데이터 모델이 다른 테이블(`cms_pages`/FAQ/팝업/배너 vs `boards`/`board_posts`)이므로 한 표의 셀 값을 다른 표에 그대로 적용하지 않는다. 예를 들어 §9에서 Compliance Admin이 약관(CMS)에 △(등록/수정) 권한을 갖는다는 표기가 Board Engine의 어떤 게시판에도 자동으로 적용되는 것은 아니다 — 두 모듈은 혼동하지 않는다.

## 28. 공제조합 연동 관리

PRD.md §5.68(한국 공제조합 연동 보강), DATABASE.md §3.59(D-075 — §3.16 확장)를 포함. 관련 테이블: 신규 `compliance_member_registrations`(회원별 공제번호/공제증서)/`compliance_transmission_items`(회원/후원관계/매출/수당/환불/반품/취소 항목별 전송상태), 연동 등록은 신규 테이블 없이 기존 `external_api_connections`(§13 API Center, `category='COMPLIANCE_REPORTING'`)를 재사용, 호출 로그는 기존 `external_api_call_logs`를 재사용한다. **본 모듈은 Tenant별 선택 기능이다** — Tenant가 직접판매공제조합/한국특수판매공제조합 연동을 켤지 말지 선택하며, 이는 7역할 프레임 자체를 바꾸지 않는다(신규 최상위 역할 없음).

1. **연동 설정(`external_api_connections`, category=COMPLIANCE_REPORTING) 자체의 삭제는 §13(API Center)과 동일한 보안 추정 논리를 적용해 SuperAdmin 한정으로 보았으나, 인증키(`auth_key_ref`) 등 시스템 보안 핵심 영역의 물리 삭제 허용 여부는 §13 자체가 명시하지 않아 미확정에 준한다.** 회원 등록 정보(`compliance_member_registrations`)의 삭제는 append-only 성격의 `compliance_transmission_items`(전송 이력)와 연결되어 있어 일반적으로 허용되지 않을 가능성이 높음 — 정확한 정책은 미확정.
2. 연동 설정(API Key/Tenant별 사용여부) 자체에는 승인 워크플로우가 없다(§13 API Center 각주와 동일 — DATABASE.md §3.59가 등록을 단순 `status`(TESTING/ACTIVE/INACTIVE) 전환으로만 기술) — "승인"/"반려"는 연동 설정 레벨에 해당없음. 회원별 등록(`compliance_member_registrations.status`의 등록대기/등록완료/등록실패/해지)도 공제조합 측 응답에 따라 시스템이 갱신하는 상태이며 내부 승인 절차가 아니라고 추정 — 미확정이나 보수적으로 해당없음에 준해 표기.
3. "일괄처리"는 재전송/수동전송(`compliance_transmission_items`의 다건 재시도, DATABASE.md §3.59 "관리 기능" 참조)을 의미한다. SuperAdmin은 O로 표기 — 법무 성격상 Compliance Admin도 재전송을 트리거할 개연성이 있어 △로 표기했으나 정확한 권한 분배는 미확정(O-042).
4. **Compliance Admin은 "법무/컴플라이언스 전담 관리자"(PRD.md §5.6.4)이며, 공제조합 연동은 그 정의상 본 역할의 핵심 업무 영역일 개연성이 매우 높다** — §21(Audit Center) 각주2·§17(Report Builder) 각주1과 동일한 논리를 적용해 조회/출력/다운로드/엑셀은 O로 표기했다. 다만 연동 설정(API Key 등록, §13의 보수적 SuperAdmin 단독 원칙) 자체는 §13과의 정합성을 위해 X로 유지하고, 회원 공제번호/증서 등록·수정은 컴플라이언스 업무 범위에 들어올 개연성이 있으나(법무 전담이 공제조합 측 서류 업무를 직접 수행할 가능성) 확정 근거가 없어 △로 표기했다(O-042 적용 범위). 재전송/수동전송(일괄처리)도 동일하게 △ — 모니터링은 명백히 컴플라이언스 업무이나 실제 재전송 트리거 권한까지 갖는지는 미확정.
5. CountryAdmin은 "본인 국가의 회원만 접근"(PRD.md §5.6.4 확정) 원칙에 따라 본인 국가 회원의 공제번호/전송상태 조회 개연성이 있어 △로 표기했으나(§1 회원관리 각주4와 동일 논리), 연동 설정(API Key, Tenant 단위 전역 설정)에는 §13 정합성에 따라 접근하지 않는다(X). 등록/수정도 동일하게 X — 공제조합 등록은 회원 개인정보·법적 신고 사항이라 일반 회원관리 등록/수정과 분리해 보수적으로 표기했다(O-042 해소 시 재검토 가능).
6. **"공제조합관리자"라는 명칭은 PRD.md §5.68 본문에 직접 등장하지 않는다.** 다만 §24 각주4·§25 각주5와 동일한 패턴으로, 운영 규모가 커지면 공제조합 연동(전송대상조회/전송상태/재전송/전송로그/보고서 다운로드)을 전담하는 기능별 관리자가 생길 개연성이 있다 — 이 경우에도 PRD.md §5.6.4가 예시로 든 "정산담당/법무담당/고객지원/물류담당"과 동일한 성격의 **기능별 관리자 하위유형 후보**로 분류되어야 하며, 신규 최상위 역할을 추가하지 않는다. O-042(기능별 관리자 역할 목록·권한 매핑 미확정) 해소 시 함께 결정 권고 — §33 항목 12 참조.
7. Partner는 본인의 공제번호/등록상태(`compliance_member_registrations`에서 본인 `member_id` 행)만 조회 가능(있다면) — 출력/다운로드도 본인 공제증서(`certificate_file_ref`)에 한정된다고 추정. 타 회원의 공제 정보, 전송 항목(`compliance_transmission_items`), 연동 설정에는 어떤 형태로도 접근하지 않는다 — 등록/수정은 회원이 직접 하지 않고 관리자가 처리하는 절차로 추정되어 X(정확한 신청 흐름의 존재 여부는 PRD.md §5.68에 명시 없어 미확정에 가까우나, 다른 회원 민감정보 등록과 마찬가지로 보수적으로 X 표기).

## 29. E-Wallet 관리

PRD.md §5.69(전자지갑), DATABASE.md §3.60(D-075)을 포함. 관련 테이블: 신규 `member_wallets`(회원별·통화별 지갑)/`wallet_transactions`(append-only 원장)/`wallet_withdrawal_requests`(출금 신청, 승인은 기존 Workflow Engine §12를 `subject_type='WALLET_WITHDRAWAL'`로 재사용). **본 모듈은 Tenant별 선택 기능이며, §5(정산관리)와 별개의 신규 워크플로우다 — §5의 정산 승인 구조는 본 절로 인해 어떤 형태로도 수정되지 않는다(SETTLEMENT-RULES.md §9 변경 금지 원칙 유지).** 지갑 적립(`wallet_transactions.transaction_type='EARN'`)은 정산이 이미 확정한 `settlement_items` 금액을 인용하는 추가 지급 채널일 뿐이며, 후원수당 1차 산정·35% 한도 검증·세금 계산(§5 각주 전체)에는 영향을 주지 않는다.

1. `wallet_transactions`는 append-only 원장이므로(DATABASE.md §3.60, §5 정산관리 각주1과 동일 원칙) 일반적 의미의 "등록"(신규 거래 생성)은 시스템(배치)·관리자(수동 보정)가 새 원장 행을 추가하는 것을 의미하며, "수정"·"삭제"는 기존 원장 행에 대해서는 존재하지 않는다(append-only). 지갑(`member_wallets`) 자체의 "등록"은 회원 가입/최초 거래 시 시스템이 자동 생성하는 것으로 추정되며 관리자의 수동 등록 개연성은 낮으나 확정 근거 없음 — △로 표기. 지갑 "삭제"는 회원의 거래 이력을 지우는 것과 같아 일반적으로 허용되지 않을 가능성이 높음(X).

CountryAdmin은 "본인 국가의 회원만 접근"(PRD.md §5.6.4) 원칙에 따라 본인 국가 회원의 지갑/거래내역 조회 개연성을 △로 표기했으나(§1 회원관리 각주4·§5 정산관리 각주3과 동일 논리), 출금 승인·반려는 각주3에 따라 SuperAdmin/Compliance Admin에 한정되므로 CountryAdmin은 X로 명시했다 — 일반 정산 승인조차 CountryAdmin에게 부여되지 않은 §5의 기존 보수적 원칙(미확정이지만 표에는 명시적으로 부여하지 않음)과의 정합성을 위해, 실제 자금이 나가는 출금 승인은 더 보수적으로 X 확정에 가깝게 표기했다.
6. **"지갑관리자"라는 명칭은 PRD.md §5.69 본문에 직접 등장하지 않으며, "관리자"라고만 표현한다.** §29 관리 기능(회원 지갑 조회/거래내역 조회/출금신청 관리/출금 승인/출금 반려/출금 완료/수동 보정/지갑 잠금/지갑 해제/통화별 통계, PRD.md §5.69)을 전담하는 기능별 관리자가 향후 분리될 개연성은 §24/§25/§28과 동일한 패턴이다 — 신규 최상위 역할이 아니라 기능별 관리자 하위유형 후보로만 분류되며, O-042 해소 시 함께 결정 권고. §33 항목 12 참조.

6. **Partner는 본 절(관리자의 PG/Webhook 운영 화면)에 어떤 형태로도 접근하지 않는다 — (해당없음)으로 표기했다.** 회원이 쇼핑몰에서 결제수단을 선택하고 결제를 진행하는 행위 자체는 §8(쇼핑몰) 각주4(주문 등록/결제) 및 §3(주문관리) 각주4(본인 주문 조회)가 이미 다루는 기존 결제 흐름이며, 글로벌 PG 추가로 결제수단 목록이 늘어나는 것은 회원 입장에서는 기존 결제 UX의 확장일 뿐 새로운 권한 표면이 아니다 — 별도 셀을 신설하지 않는다. "삭제"만 X로 표기한 것은 결제수단 등록(`payment_methods`, §8/§3 기존 권한 범위)과의 일관성을 위함이며, 본 절(PG 연동/Webhook 운영) 차원에서는 Partner에게 그 자체가 의미 없는 액션이다.

> a. **Notification Inbox(§5.75, `notification_inbox_states`, recipient_type=ADMIN)는 본인 수신함만 조회·처리할 수 있다 — 타인 수신함은 어떤 역할도 조회할 수 없다.** 이는 본인 데이터 한정 원칙이며 위 표에 별도 행을 두지 않는다 — `notification_inbox_states`는 기존 `notifications`(§22 Notification Center) 위에 얹는 읽음/보관/고정 상태 오버레이일 뿐 신규 발송 권한이 아니므로, "조회/수정"(읽음 처리 등)은 본인 행에 한정되어 7역할 모두 동일하게 O(본인 한정)다 — SuperAdmin이라고 해서 타 관리자의 수신함을 대신 조회할 수 있는 것이 아니다(예외 없음, 강조).
> b. **Tenant Usage Dashboard(§5.76)는 Dashboard Builder(§16)의 위젯 추가일 뿐 신규 데이터 모델이 없다** — 위젯 등록/수정/삭제 권한은 §16의 기존 권한 구조(§16 각주2와 동일 정합성 원칙)를 그대로 따른다. 위젯이 노출하는 사용량 데이터의 스코프는 CountryAdmin은 본인 국가(Tenant 단위 집계가 도입될 경우 그 산하 국가)만, SuperAdmin/Compliance Admin은 전체를 본다는 일반 원칙(§16 각주1과 동일)을 적용한다 — 다만 본 플랫폼의 현재 7역할 목록(§5.6.4)에는 "Tenant 관리자"라는 명칭의 역할이 존재하지 않으며(§20 Multi-Tenant 설정 각주1 참조, Multi-Tenant 활성화 자체도 D-035로 보류 상태), Tenant 단위 사용량 집계가 정확히 어떤 기존 역할(CountryAdmin 또는 향후 도입될 기능별 관리자 하위유형)에 매핑되는지는 미확정(O-042 적용 범위)이다.
> c. **Approval History(§5.77)는 승인/반려 이력 조회 전용 화면이며, §21(Audit Center)과는 명시적으로 분리된 별개 권한이다.** Audit Center(§21)는 시스템 전반의 포렌식/책임추적 로그(`audit_logs`)를 다루는 반면, Approval History는 Approval Center(위 본문 각주4)가 처리한 승인 항목의 결과만 보여주는 좁은 범위의 업무 화면이다 — 따라서 Approval History 조회 권한은 §21의 조회 권한(SuperAdmin/Compliance Admin O, CountryAdmin △, 그 외 미확정 또는 X)을 그대로 상속하지 않으며, 대신 본문 각주4가 정의한 "각 승인 항목이 속한 원래 모듈의 기존 승인 권한 보유자는 그 항목의 History도 조회 가능"이라는 더 넓은 원칙을 따른다고 추정한다(예: CountryAdmin은 §21에서는 Audit Center 조회가 △·미확정 수준이지만, 본인이 처리 권한을 가진 본인 국가 스코프 승인 항목의 History는 조회 가능할 개연성이 높음). 정확한 경계는 미확정(O-042).
> d. **Personal Workspace(§5.78)·Recent Activity(§5.74)는 My Dashboard+Favorite Menu+Recent Activity+Quick Action을 한 화면에 모은 통합 화면이며 신규 데이터 모델이 없다** — §16(Dashboard Builder) 각주3의 "My Dashboard는 신규 권한 구분을 필요로 하지 않는다" 원칙과 동일하게, Personal Workspace 자체는 위 본문 각주1~3이 이미 정의한 각 구성요소(Favorite Menu 각주2, Recent Activity는 본인의 과거 액션 federated 조회)의 권한을 합쳐 보여줄 뿐이다 — **Recent Activity는 관리자 본인이 과거에 수행한 액션만 federated 조회하므로(§1 회원관리 각주8 Customer Timeline과 동일한 federated 조회 패턴), 타 관리자의 활동 이력은 절대 노출하지 않는다(본인 한정, 예외 없음).** 별도 행·별도 셀을 신설하지 않는다.
> e. **Command Palette(§5.81)·Universal Clipboard(§5.82)는 DB 영향이 없는 순수 프론트엔드 기능이며, 권한도 N/A(해당없음)다.** Command Palette는 위 각주1의 Global Search와 동일 원칙(기존 모듈 권한 그대로 인용)을 따르는 라우팅 단축키이고, Universal Clipboard(여러 화면 간 복사한 값을 임시 보관해 붙여넣기를 돕는 프론트엔드 전용 기능)는 서버에 어떤 신규 테이블도 두지 않으며 권한 검증 대상 자체가 아니다 — 본 매트릭스가 다루는 7역할×11액션 어디에도 매핑되지 않는다.

## 33. 본 매트릭스에서 새로 발견된 미확정 항목 (추가 확인 필요)

아래 항목은 기존 DECISIONS.md Open Decision에 없거나, 기존 항목(O-042/O-066)의 범위를 모듈별로 구체화하는 과정에서 새로 드러난 세부 공백이다. **DECISIONS.md는 수정하지 않았으며**, 본 섹션은 향후 사업팀이 O-042/O-066 후속 라운드 또는 신규 Open Decision으로 처리할 것을 권고하는 목록이다.

1. **조직 이동 이력/통계의 조회 권한 범위** — PRD.md §5.16.6은 "조직 이동 이력 조회"를 필수 기능으로 명시하지만, 일반(예약형) 조직 이동 권한이 없는 CountryAdmin 등이 자기 국가 범위 내 조직 이동 이력만이라도 조회할 수 있는지는 §5.16.5(권한)에 별도 언급이 없다. 본 매트릭스는 보수적으로 전면 X 처리했으나, 운영상 CountryAdmin에게 "조회만" 허용할 필요가 있는지는 검토 권고.
2. **API Center / Scheduler Center / System Settings의 역할별 접근권** — 이 3개 ERP Core 시스템 운영 모듈은 PRD.md에 SuperAdmin 외 역할의 접근 가능성이 전혀 언급되어 있지 않다. 본 매트릭스는 가장 보수적으로 SuperAdmin 단독 접근으로 표기했는데, 이는 실제 운영(예: 시스템 운영 담당이라는 기능별 관리자가 API Center에 접근해야 하는 상황)과 충돌할 수 있어 O-042(기능별 관리자 권한 매핑) 해소 시 함께 다룰 필요가 있다.
3. **Compliance Admin의 정확한 액션 단위 권한** — O-066이 "권한 범위 및 임명 절차"를 미확정으로 이미 추적하고 있으나, 본 매트릭스 작성 과정에서 구체적으로 "회원관리 승인/반려", "정산 승인/반려", "CMS 약관 등록/수정" 등 모듈별 세부 액션 단위로 나눠보니 단순 "조직 이동만 가능"이라는 §5.16.5 외에는 근거가 거의 없다는 점이 드러났다. O-066 해소 시 모듈별 액션 표(본 문서 형식)로 답을 받는 것을 권고.
4. **CenterAdmin의 거의 전 모듈 미확정** — 센터 구조 도입 자체가 미확정(PRD.md §5.9)이라는 선결 조건 때문에 CenterAdmin 행은 거의 모든 모듈에서 미확정으로 남았다. 센터 구조 도입이 확정되는 시점에 본 매트릭스 전체를 일괄 재검토해야 한다.
5. **CRM Center "승인" 액션의 실체** — §10(CRM)에서 "승인" 액션이 실제로 존재하는지(Follow-up 상태 전환이 승인 개념에 해당하는지)가 PRD.md §5.40에 명시적으로 나와 있지 않다 — CRM Center 화면 설계 단계에서 액션 정의 자체를 재확인할 필요.
6. **File Manager의 폴더/파일 단위 "권한관리" 세부** — PRD.md §5.32는 "권한 — 폴더/파일 단위 접근 제어(Document Center §3.18의 공개범위 패턴 재사용)"라고만 되어 있어, 이 권한관리 액션을 누가(어떤 역할이) 수행할 수 있는지 자체가 명시되어 있지 않다.
7. **Dashboard Builder/Report Builder가 노출하는 데이터의 스코프 상속 규칙** — 위젯/보고서가 각 업무 모듈의 집계 데이터를 소비할 때(§5.28.1 의존 원칙), 그 데이터가 조회자의 국가/센터 스코프를 자동으로 상속하는지(예: CountryAdmin이 위젯을 만들면 자동으로 본인 국가 데이터만 보이는지)가 PRD.md에 명시되어 있지 않다 — O-111(Dashboard Builder API 계약)과 함께 해소 필요.
8. **SEO/Feed/Analytics/Search Console/Merchant 관리자의 기능별 관리자 하위유형 분리 여부** — §24(SEO/공유이미지 관리)·§25(Digital Marketing 연동/Feed 관리)을 작성하는 과정에서, PRD.md §5.46~§5.50은 일관되게 "관리자"라고만 표현하고 SEO/Feed/Analytics/Search Console/Merchant 운영을 전담하는 별도 역할을 한 번도 명시하지 않는다는 점이 드러났다. 이는 기존 O-042(기능별 관리자 역할 목록·권한 매핑 미확정)가 이미 추적하던 공백의 모듈별 구체 사례이며, 신규 Open Decision으로 등록하지 않았다 — "정산담당/법무담당/고객지원/물류담당"과 동일한 선상에서 "SEO 관리자/Feed 관리자/Analytics 관리자/Search Console 관리자/Merchant 관리자"가 향후 기능별 관리자의 명시적 하위유형으로 추가될지, 아니면 상품관리(§2)·API Center(§13)·CMS(§9) 등 기존 모듈 담당자가 겸임할지는 O-042 해소 시 함께 결정 권고. 참고로 §24 각주4·§25 각주5에서 이미 이 후보 명칭들을 기능별 관리자 하위유형 후보로만 다루었을 뿐 신규 최상위 역할로 만들지 않았다.
9. **D-070 / O-195가 DECISIONS.md에 아직 등록되지 않음** — PRD.md는 상단 상태줄과 §5.47~§5.50 헤더에서 "D-070"을 인용하고, §5.48 본문(검색엔진 지표 캐시 저장 여부)에서 "신규 Open Decision O-195"를 등록한다고 서술하지만, 실제로 DECISIONS.md의 결정 로그와 Open Decision 표에는 D-070 항목도 O-195 행도 존재하지 않는다(DECISIONS.md는 D-069/O-194까지만 등록되어 있음). 본 매트릭스는 작업 지시에 따라 "D-070"을 라운드 명칭으로만 인용했을 뿐 DECISIONS.md를 직접 수정하지 않았다 — PRD.md와 DECISIONS.md 간 동기화(D-070 결정 항목 및 O-195 신설)는 사업팀이 DECISIONS.md 쪽에서 별도로 처리할 필요가 있다.
10. **D-072 / O-197~O-199가 DECISIONS.md에 아직 등록되지 않음 (바로 위 항목 9와 동일한 패턴) — 그리고 "운영 Queue 처리"의 기능별 관리자 하위유형 분리 여부**: DATABASE.md §3.57은 상단 헤더와 본문에서 "D-072"·"O-197"(장바구니 "나중에 구매하기" 처리)·"O-198"(가격알림 트리거 기준)·"O-199"(저장된 검색조건 도입 여부)를 이미 인용하지만, DECISIONS.md에는 이 라운드의 결정 로그(### D-072 항목)도 Open Decision 표의 O-197/O-198/O-199 행도 아직 존재하지 않는다(DECISIONS.md는 D-071/O-196까지만 등록되어 있고, D-071 본문은 "본 라운드를 끝으로 설계는 최종 종료되고 이후 신규 요구사항은 Change Request(CR) 방식으로만 관리한다"고 명시한 상태다) — 본 매트릭스는 작업 지시에 따라 "D-072"/"O-197"/"O-198"/"O-199"를 그대로 인용했을 뿐 DECISIONS.md를 직접 수정하지 않았으며, 이 동기화(D-072 결정 항목 신설 및 D-071의 "설계 최종 종료" 선언과의 관계 정리)는 사업팀이 별도로 처리할 필요가 있다. 별개로, §26(운영자 대시보드/관리자 업무 Queue) 작성 과정에서 "운영 Queue 처리"(결제실패확인/무통장입금확인/송장등록필요/배송지연확인/반품·교환·환불 승인대기/리뷰신고처리/상품승인대기/재고부족처리/CS문의미답변 횡단 조회)를 전담하는 별도 기능별 관리자 하위유형("운영 Queue 담당")이 필요한지, 아니면 각 Queue 항목이 속한 모듈의 기존 담당자(정산담당/물류담당/고객지원 등, PRD.md §5.6.4 예시)가 그대로 처리하는 것으로 충분한지가 PRD.md §5.56~§5.61에 명시되어 있지 않다는 점이 드러났다 — 이는 §24 각주4·§25 각주5와 동일한 성격의 O-042(기능별 관리자 역할 목록·권한 매핑 미확정) 적용 범위의 모듈별 구체 사례이며, 신규 Open Decision으로 등록하지 않았다. O-042 해소 시 함께 결정 권고.
11. **Board Engine 댓글(`board_post_comments`) 모더레이션 정책의 부재** — §27(Board Engine) P5를 작성하는 과정에서, PRD.md §5.67은 "댓글/답글"을 `feature_flags`의 ON/OFF 토글 항목으로만 언급하고(§5.67.4) 댓글 모더레이션(신고 누적 시 자동 비노출, 비속어 필터링, 관리자 일괄 삭제 기준 등) 정책 자체는 명시하지 않는다는 점이 드러났다. 본 매트릭스는 "회원은 본인 댓글만 수정·삭제, 관리자는 전체 모더레이션 가능"이라는 일반 원칙만 표기했을 뿐(§27 P5), 구체적 모더레이션 기준·신고 처리 흐름은 미확정이다. 이 공백은 §8(쇼핑몰) 각주1·§26 각주3이 이미 추적 중인 "회원 게시물(리뷰 등) 모더레이션 정책 미확정"과 동일한 성격이며, board 댓글로 그 범위가 한 번 더 확장된 사례다 — 신규 Open Decision으로 등록하지 않았다. 참고로 기존 CMS(§9)에는 댓글 기능 자체가 없어 선례가 없으며, O-200(기존 CMS 콘텐츠의 Board Engine 통합 마이그레이션 여부)이 향후 통합 방향으로 해소될 경우 댓글 모더레이션 정책도 함께 정리될 필요가 있다. O-042(기능별 관리자, "고객지원" 담당 후보) 해소 또는 §27 화면 설계 단계에서 함께 결정 권고.
12. **"지갑관리자"/"공제조합관리자" 기능별 관리자 하위유형 분리 여부** — §28(공제조합 연동 관리)·§29(E-Wallet 관리)를 작성하는 과정에서, PRD.md §5.68/§5.69 본문이 일관되게 "관리자"라고만 표현하고 공제조합 연동·E-Wallet 운영(회원 지갑 조회/거래내역 조회/출금신청 관리/출금 승인/수동 보정/지갑 잠금·해제/통화별 통계, 그리고 공제조합의 전송대상조회/전송상태조회/재전송/전송로그/보고서 다운로드)을 전담하는 별도 역할을 한 번도 명시하지 않는다는 점이 §24 각주4·§25 각주5·§28 각주6·§29 각주6과 동일한 패턴으로 다시 드러났다. 이는 기존 O-042(기능별 관리자 역할 목록·권한 매핑 미확정)가 이미 추적하던 공백의 모듈별 구체 사례이며, 신규 Open Decision으로 등록하지 않았다 — "정산담당/법무담당/고객지원/물류담당"과 동일한 선상에서 "지갑관리자"/"공제조합관리자"가 향후 기능별 관리자의 명시적 하위유형으로 추가될지, 아니면 정산담당(자금 흐름이라는 성격상)·법무담당(컴플라이언스 신고라는 성격상)이 겸임할지는 O-042 해소 시 함께 결정 권고. 별개로, E-Wallet의 출금 승인(§29 각주3)이 SuperAdmin/Compliance Admin에 한정된 것은 §5(정산관리)의 기존 보수적 원칙을 참고해 독자적으로 적용한 것이며, O-201(정산↔지갑 적립 분배 정책)·O-202(포인트↔지갑 전환 여부)·O-203(쇼핑몰 결제 시 지갑·포인트 우선순위)이 해소되어 지갑의 실제 운영 비중이 커지면 출금 승인 권한 배분도 함께 재검토가 필요할 수 있다(O-201~O-203은 이미 DECISIONS.md에 D-075 신규 항목으로 등록되어 있으므로 본 항목에서는 재등록하지 않고 인용만 한다).
13. **승인 위임 시 권한 레벨 검증 방식** — §32(ERP 운영 생산성 및 관리자 UX 완성) 각주4를 작성하는 과정에서, `approval_delegations`(DATABASE.md §3.62)가 "본인 위임설정"과 "관리자 대리설정"을 구분해야 한다는 점은 도출했으나, 위임 가능 범위가 동일 역할 내로 한정되는지(예: CountryAdmin은 다른 CountryAdmin에게만 위임 가능한지) 및 위임받는 쪽의 권한 레벨이 위임하는 쪽보다 낮을 경우(예: 기능별 관리자가 CountryAdmin의 승인 권한을 위임받는 경우) 이를 차단할 검증 로직이 필요한지는 PRD.md §5.72에 명시되어 있지 않다 — 이는 O-207(`approval_delegations`의 위임 가능 범위·권한 레벨 검증 방식 미확정)로 오케스트레이터가 이미 등록한 항목이며, 본 매트릭스는 §32 각주4에서 (a)/(b) 구분만 표기했을 뿐 세부 검증 로직을 고정하지 않았다 — 중복 등록하지 않고 O-207 해소 시 §32를 함께 갱신할 것을 권고한다.
