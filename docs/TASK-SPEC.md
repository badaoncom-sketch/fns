# TASK-SPEC.md — AI 작업 지시서 표준 형식

> 상태: Draft v0.3 (D-061 반영 — §6에 ERP UX Standard 컴포넌트 재사용 원칙 추가) · 최종 수정일: 2026-06-25 · 단계: 설계(Design)
> 목적: Claude Code / Codex에게 작업을 위임할 때 사용하는 표준 작업 지시서 형식을 정의한다.

## 1. 원칙

1. **한 작업 = 한 명확한 목표.** 여러 모듈을 동시에 변경하는 작업은 가능한 한 분리한다.
2. **문서가 먼저.** 구현 전 관련 규정 문서([COMPENSATION-RULES.md](COMPENSATION-RULES.md) 등)가 이미 확정되어 있어야 하며, 미확정 상태에서는 구현 작업을 발행하지 않는다.
3. **현재 단계는 설계(Design)다.** 별도 승인 전까지 모든 작업 지시는 "문서 작성/수정" 범위로 한정하며, 코드 생성을 요청하지 않는다.
4. 모든 작업 지시는 [DO-NOT-TOUCH.md](DO-NOT-TOUCH.md)를 위반하지 않아야 한다.
5. **구현 단계 진입 후에는, 모든 작업 지시가 5개 서비스(web/api/worker/scheduler/redis — [ARCHITECTURE.md](ARCHITECTURE.md) §1~§2) 중 어디에 속하는지 명시해야 한다.** §6 참조.

## 2. 작업 분류 체계

| 태그 | 의미 |
|---|---|
| [설계] | 문서 작성/수정, 규칙 정의 |
| [구현] | 코드 작성 (설계 단계 종료 후 사용) |
| [버그수정] | 기존 구현의 결함 수정 (구현 단계 이후 사용) |
| [리팩토링] | 동작 변경 없는 구조 개선 (구현 단계 이후 사용) |
| [문서] | 문서 정리/포맷/오타 등 경미한 수정 |
| [리뷰] | 기존 산출물에 대한 검토 요청 |

## 3. 작업 지시서 템플릿

```
## 제목
[태그] 한 줄 요약

## 배경
이 작업이 왜 필요한가. 관련 문서/이전 결정 참조.

## 목표
이 작업이 끝나면 달성되어야 하는 상태 (가능한 한 구체적으로).

## 범위
- In-scope: 이 작업에서 다루는 것
- Out-of-scope: 이 작업에서 다루지 않는 것 (명시적으로)

## 참고 문서
- 이 작업이 전제하는 docs/ 문서 목록

## 서비스 배치 (구현 단계 — §6)
- 이 작업의 코드가 속하는 서비스: web / api / worker / scheduler / redis 설정 중 하나
- (api인 경우) Job 생성만 하는지 확인, 무거운 계산 포함 여부 — 포함되면 작업을 재정의

## 제약사항
- DO-NOT-TOUCH.md 중 해당되는 항목
- 기타 제약

## 완료 조건 (Definition of Done)
- 구체적으로 확인 가능한 조건 목록

## 산출물
- 변경/생성될 파일 목록
```

## 4. 작업 진행 중 문서 갱신 규칙

- 작업 결과로 문서 내용이 바뀌면 해당 작업 완료 시 [CHANGELOG.md](CHANGELOG.md)에 한 줄 기록한다.
- 작업 결과로 이전에 "미확정"이던 항목이 확정되면 해당 문서를 갱신하고 [DECISIONS.md](DECISIONS.md)에 결정 사항을 기록한다.
- 작업 중 새로운 미확정 사항이 발견되면 해당 문서의 "Open Decisions" 섹션과 [DECISIONS.md](DECISIONS.md)에 추가한다.

## 5. 현재 단계 정책

- **코드 생성 금지.** 설계 단계 종료([DECISIONS.md](DECISIONS.md)의 Open Decisions 해소 + PRD/ARCHITECTURE/DATABASE 합의)가 명시적으로 선언되기 전까지, 모든 작업 지시는 `docs/` 내 문서 작성/수정에 한정한다.
- 설계 단계 종료 선언 이후에도, 구현 작업은 본 문서의 템플릿을 따르는 작업 지시서 단위로 발행되어야 한다.

## 6. 서버 구조 배치 원칙 (구현 단계 적용 — [DECISIONS.md](DECISIONS.md) D-010)

FNS의 서버는 Railway 프로젝트 1개 안에 **web/api/worker/scheduler/redis** 5개 서비스로 확정되어 있다 ([ARCHITECTURE.md](ARCHITECTURE.md) §1~§2). 구현 단계 작업 지시서는 다음을 반드시 따른다.

1. **모든 구현 작업 지시서는 "서비스 배치"를 명시한다** (§3 템플릿 참조) — 어느 서비스(web/api/worker/scheduler)의 코드를 만드는 작업인지 모호하면 작업을 발행하지 않는다.
2. **web에 계산 로직을 넣는 작업은 발행하지 않는다.** web 관련 작업은 UI/조회 표시로 범위를 한정한다.
3. **api 작업에 무거운 계산(후원수당/정산/세금/프로모션 판정/보고서 생성)을 포함시키지 않는다.** api 작업은 "요청 검증 + Job 생성 + 조회"로 범위를 한정하고, 계산 자체는 별도의 worker 작업으로 분리해 발행한다.
4. **worker 작업 지시서는 입력(Job payload)과 출력(기록 대상 Supabase 테이블)을 명시한다.** worker는 Redis가 아닌 Supabase PostgreSQL에 최종 결과를 기록해야 한다.
5. **scheduler 작업 지시서에는 계산 로직을 포함하지 않는다.** scheduler 작업은 "Job 트리거(cron 등록)"로 범위를 한정한다.
6. 위 원칙을 위반하는 작업 지시서는 발행 전 반드시 재정의한다 — [DO-NOT-TOUCH.md](DO-NOT-TOUCH.md)에 동일 원칙이 금지 항목으로 명시되어 있다.
7. **ConfirmDialog/Toast/LoadingButton 등 ERP UX Standard 컴포넌트([PRD.md](PRD.md) §5.44, [DECISIONS.md](DECISIONS.md) D-061) 작업 지시서는 화면별로 새로 만들지 않고 공통 컴포넌트 라이브러리를 재사용해야 한다** — 특정 화면에서 다른 UX가 필요하면 작업을 발행하기 전에 표준 자체를 먼저 갱신한다(§5.44.12 원칙).
