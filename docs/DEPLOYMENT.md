## 0. 문서의 성격

- 본 문서는 아직 코드/운영 환경이 없는 **설계 단계**에서 작성하는 **권장 절차안**이다. 실제 구현 단계에 진입하면 [DECISIONS.md](DECISIONS.md)에서 관련 Open Decision이 확정되는 대로 본 문서를 갱신한다.
- 본 문서가 인용하는 Open Decision(O-037, O-144, O-148, O-022, O-025, O-161)은 **모두 미확정**이며, 아래 내용 중 이 항목들과 관련된 서술은 "확정"이 아니라 "권장안"으로 읽어야 한다.
- [DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §1.3(서버 구조)/§1.4(Git/배포)의 원칙을 위반하는 절차는 본 문서에 포함하지 않는다 — 5개 서비스(web/api/worker/scheduler/redis) 구조, append-only 원장 원칙, force push·hook 우회 금지 원칙은 그대로 전제한다.

## 1. 환경 구조 (권장안 — 확정은 O-148 참조)

> **O-148(Railway 환경 dev/staging/prod 분리 개수·명명 및 Supabase 프로젝트 분리 여부)은 미확정이다.** 아래는 일반적인 3단계 구조를 권장안으로 제시하는 것이며, 실제 채택 여부·환경 개수·명칭은 O-148이 확정된 후 따른다.

### 1.1 권장 3단계 구조

| 환경 | 목적 | Railway 프로젝트(권장) | Supabase 프로젝트(권장) |
|---|---|---|---|
| Development | 기능 개발, 통합 테스트 | 별도 Railway 프로젝트(또는 동일 프로젝트 내 별도 environment) | 별도 Supabase 프로젝트 — **실 회원 PII/실 정산 금액 데이터는 절대 복제하지 않는다**([DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §1.2). 가상/목업 데이터만 사용 |
| Staging | 운영 배포 전 최종 검증, 회귀 테스트 | 별도 Railway 프로젝트(또는 별도 environment) | 별도 Supabase 프로젝트 — 마이그레이션을 prod에 적용하기 전 동일 스키마로 먼저 검증하는 용도 |
| Production | 실 서비스 운영 | 별도 Railway 프로젝트 | 별도 Supabase 프로젝트 — 유일한 source of truth([ARCHITECTURE.md](ARCHITECTURE.md) §2.6) |

- **권장 원칙**: 3개 환경은 각각 독립된 Supabase 프로젝트를 사용한다(스키마는 동일, 데이터는 분리). 단일 Supabase 프로젝트를 스키마/RLS 정책만으로 환경 분리하는 방식은 PII 격리가 약해질 위험이 있어 권장하지 않는다 — 다만 이는 권장안이며 최종 결정은 O-148에서 비용/운영 부담과 함께 재검토한다.
- 환경별로 [ARCHITECTURE.md](ARCHITECTURE.md) §1의 5개 서비스(web/api/worker/scheduler/redis) 구조를 동일하게 복제한다 — 환경마다 구조가 달라지지 않는다([DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §1.3 "5개 서비스 구조를 임의로 합치거나 쪼개지 않는다"는 원칙은 환경 간에도 동일하게 적용).
- 환경 간 전파 순서(권장): Development → Staging → Production. Staging을 건너뛰고 Development에서 바로 Production으로 배포하지 않는다.
- scheduler는 환경별로 정확히 1개 인스턴스만 떠 있어야 한다([ARCHITECTURE.md](ARCHITECTURE.md) §9.1) — 특히 Staging/Production에서 scheduler가 중복 기동되면 정산/조직이동 적용 Job이 중복 생성될 수 있으므로, 인스턴스 수를 1로 고정하는 배포 설정을 권장한다.

### 1.2 환경 변수/시크릿 관리 (권장안)

- 환경별로 Railway 환경변수를 분리 관리하며, `.env`/DB 자격증명/Railway·Supabase API 키는 레포에 커밋하지 않는다([DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §1.2).
- Production 환경변수에 대한 접근 권한은 최소 인원(예: SuperAdmin 역할에 준하는 운영 책임자)으로 제한하는 것을 권장한다 — 구체적 정책은 [DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §3 "향후 추가 예정"의 "프로덕션 환경 변수 목록 및 접근 권한 정책" 항목에서 후속 확정한다.

## 2. 배포 순서

### 2.1 배포 전 체크리스트 (권장안)

1. 변경 사항이 [DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §1.2/§1.3 원칙(계산 로직 위치, append-only 원장, 5서비스 경계)을 위반하지 않는지 확인.
2. 코드 리뷰/CI 통과 여부 확인(테스트 도구·CI 파이프라인 구체 사양은 별도 Open Decision 대상이며 본 문서는 절차만 기술).
3. 이번 배포에 DB 마이그레이션이 포함되는지 확인 — 포함된다면 §3의 마이그레이션 절차를 먼저 수행.
4. Staging에서 동일 변경 사항이 이미 검증되었는지 확인(Production 배포 직전 단계).
5. 정산/수당 계산 로직에 영향을 주는 변경인 경우, [COMPENSATION-RULES.md](COMPENSATION-RULES.md)/[SETTLEMENT-RULES.md](SETTLEMENT-RULES.md)에 확정된 내용과 일치하는지 재확인([DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §1.2).
6. 롤백 계획(§4)을 배포 전에 미리 정해 둔다 — 배포 후 즉흥적으로 결정하지 않는다.

### 2.2 서비스 배포 순서 (권장안)

[ARCHITECTURE.md](ARCHITECTURE.md) §1의 의존관계(`web` → `api` → `redis` ← `worker`/`scheduler`)를 따라, 호출을 받는 쪽이 호출하는 쪽보다 먼저 준비되어 있어야 한다는 원칙으로 다음 순서를 권장한다.

```
1. redis        — 큐/캐시 인프라. 다른 모든 서비스가 의존하므로 가장 먼저 가용해야 한다(보통 상시 가동, 무중단 운영).
2. api          — web/worker/scheduler가 호출하거나 Job을 적재하는 대상. worker/scheduler보다 먼저 배포해
                  새 API 계약(엔드포인트/스키마)이 준비된 상태를 만든다.
3. worker       — api/scheduler가 적재한 새 Job 포맷을 처리할 수 있어야 하므로 api 배포 직후 배포한다.
4. scheduler    — 새 Job 트리거 로직(cron 등록 변경 등)을 worker가 처리할 수 있는 상태가 된 뒤 배포한다.
5. web          — api가 새 응답 포맷/엔드포인트를 이미 제공하는 상태에서 마지막에 배포한다(하위 호환 깨짐 방지).
```

- **핵심 원칙**: API 계약을 변경하는 배포는 "이를 제공하는 쪽(api/worker)"을 "이를 소비하는 쪽(web)"보다 먼저 배포한다. 반대로 하면 web이 아직 존재하지 않는 엔드포인트/필드를 호출해 오류가 발생할 수 있다.
- api는 하위 호환을 깨는 변경(필드 삭제, 응답 구조 변경 등)을 도입할 때 가능하면 한 배포 사이클 동안 신/구 버전을 동시 지원하는 것을 권장한다 — web 배포가 완료될 때까지의 과도기를 안전하게 넘기기 위함.
- worker/scheduler는 [ARCHITECTURE.md](ARCHITECTURE.md) §1.1 원칙상 계산/트리거 책임이 분리되어 있으므로, 두 서비스를 항상 동시에 배포할 필요는 없으나 Job 포맷(payload 구조)을 변경하는 경우에는 worker(소비자)를 scheduler/api(생산자)보다 먼저 배포해야 한다 — 즉 "생산자가 신규 포맷을 만들기 시작하기 전에 소비자가 신규 포맷을 처리할 수 있어야 한다."
- scheduler는 인스턴스 1개 고정이 전제이므로, 배포 시 짧은 다운타임(롤링 배포 시 일시적으로 0개 또는 2개 인스턴스가 되는 구간)이 cron 누락/중복으로 이어지지 않도록 배포 플랫폼의 롤링 전략을 확인한다.

### 2.3 DB 마이그레이션 적용 시점 (권장안)

- 마이그레이션은 **코드 배포보다 먼저, 별도 단계로** 적용하는 것을 권장한다 — 코드와 동시에 묶어서 배포하지 않는다.
- 순서 권장: `(1) 하위 호환되는 마이그레이션 적용 → (2) api/worker 배포 → (3) scheduler 배포 → (4) web 배포 → (5, 필요 시) 하위 호환을 끝내는 후속 마이그레이션(컬럼 삭제 등) 적용`.
- 즉 마이그레이션 자체도 "확장(add) 먼저, 축소(drop) 나중"의 2단계로 나누는 것을 권장한다 — 컬럼/테이블을 추가하는 마이그레이션은 그것을 사용하는 코드보다 먼저 적용하고, 더 이상 쓰이지 않는 컬럼/테이블을 제거하는 마이그레이션은 그것을 참조하던 코드가 모두 내려간 뒤에 적용한다.
- Production에 마이그레이션을 적용하기 전 Staging에서 동일 마이그레이션을 먼저 실행해 검증한다(§1.1).
- 프로덕션 데이터베이스는 마이그레이션/배포 파이프라인을 거치지 않은 직접 수정(콘솔 등)을 하지 않는다([DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §1.2) — 긴급 hotfix라도 동일 경로를 따른다.

## 3. Migration

> ORM/마이그레이션 도구 선정(O-022, [DATABASE.md](DATABASE.md))은 미확정이므로, 아래는 도구 중립적인 절차로 기술한다.

### 3.1 마이그레이션 실행 절차 (권장안, 도구 중립)

1. 마이그레이션 스크립트(또는 마이그레이션 도구가 생성한 변경분)를 버전 관리에 커밋하고 코드 리뷰를 거친다.
2. Development 환경에서 먼저 적용해 스키마 변경이 의도대로 동작하는지 확인한다.
3. Staging 환경에 적용한다 — 가능하다면 Production과 유사한 데이터 볼륨/구조로 회귀 테스트를 수행한다.
4. Production 적용 직전, 해당 시점의 백업/복구 시점(§5)을 확인해 둔다(마이그레이션 자체가 실패 시 복구 기준점이 되므로).
5. Production에 적용한다 — 가능하면 트래픽이 적은 시간대에, 그리고 §2.3의 "확장 먼저, 축소 나중" 순서를 지킨다.
6. 적용 후 마이그레이션 이력(어떤 마이그레이션이 어떤 환경에 언제 적용되었는지)을 기록한다 — 이력 기록 방식은 채택할 도구가 제공하는 기본 메커니즘을 따르는 것을 권장(도구 중립 원칙상 본 문서에서 특정 메커니즘을 지정하지 않음).
7. append-only 원장 테이블(`commission_records`, `settlement_items`, `audit_logs` 등 — [DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §1.2)에 영향을 주는 마이그레이션은 특히 신중히 다룬다 — 스키마 변경(컬럼 추가 등)은 가능하나, 기존 행의 값을 일괄 `UPDATE`로 보정하는 마이그레이션은 원칙적으로 지양하고 보정 엔트리 추가 방식을 우선 검토한다.

### 3.2 마이그레이션 실패 시 대응 (권장안)

1. **즉시 배포를 중단**한다 — 마이그레이션이 실패한 상태에서 코드 배포(§2.2)를 진행하지 않는다.
2. 마이그레이션이 트랜잭션 단위로 실행 가능한 종류였다면, 실패 시 자동 롤백되었는지 확인한다(부분 적용 여부 확인).
3. 부분 적용이 의심되면, 적용된 부분과 미적용된 부분을 식별한 뒤 두 가지 중 선택한다: (a) 실패한 변경분을 수동으로 역행(reverse)하는 보정 마이그레이션을 작성해 적용 전 상태로 되돌린다, (b) 실패 원인을 수정한 새 마이그레이션을 작성해 나머지를 마무리한다. 어느 쪽이든 기존 마이그레이션 파일을 임의로 수정해 "다시 실행"하지 않는다 — 새 마이그레이션으로 대응한다.
4. Production에서 실패했고 §5의 백업/PITR로 복구가 필요한 수준이라면 §5.2 절차를 따른다.
5. 원인 분석 후 Development/Staging에서 재현·검증을 마친 다음에야 Production에 재시도한다.

## 4. Rollback

코드 롤백과 DB 마이그레이션 롤백은 **서로 다른 메커니즘이며 분리해서 다룬다.**

### 4.1 코드 롤백 (권장안)

- 배포 플랫폼(Railway)의 이전 배포본으로 되돌리는 기능을 1차 수단으로 사용한다.
- §2.2의 배포 순서를 역순으로 적용하는 것을 권장한다 — 즉 문제가 발견되면 `web` 먼저 이전 버전으로 되돌리고(사용자 영향 최소화), 이어서 `scheduler`/`worker`/`api` 순으로 검토한다. 단, api 응답 계약이 이미 변경되어 신규 web만 호환되는 상태라면 web을 먼저 내리는 것이 오히려 오류를 키울 수 있으므로, 실제로는 "어느 쪽이 하위 호환을 깨뜨렸는지"를 먼저 판단한 뒤 그 변경을 만든 서비스부터 되돌린다.
- 코드 롤백은 `main`(또는 기본) 브랜치에 대한 force push나 히스토리 재작성을 사용하지 않는다 — 새 커밋(revert)으로 이전 상태를 재현한다([DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §1.4).
- 롤백 중에도 git hook을 우회(`--no-verify` 등)하지 않는다 — 명시적 승인 없이는([DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §1.4).

### 4.2 DB 마이그레이션 롤백 (권장안)

- 마이그레이션 롤백은 코드 롤백과 **반드시 함께 계획**해야 한다 — 코드만 되돌리고 스키마는 그대로 두면, 이전 코드가 새 스키마와 호환되지 않을 수 있다(반대도 마찬가지).
- §2.3의 "확장 먼저, 축소 나중" 원칙을 지켰다면, 코드만 롤백해도 이전 스키마(아직 축소되지 않은 상태)와 호환되는 경우가 많다 — 이것이 해당 원칙을 권장하는 이유이기도 하다.
- 컬럼/테이블 삭제처럼 비가역적인 마이그레이션은 가능하면 즉시 실행하지 않고, 한 배포 사이클 이상 유예 기간을 두는 것을 권장한다 — 그래야 문제가 발견됐을 때 "삭제 전" 상태로 돌아갈 길이 남는다.
- 데이터를 변형하는 마이그레이션(예: 컬럼 값 일괄 변환)을 롤백해야 하는 경우, 단순히 반대 방향 스크립트를 실행하는 것이 항상 안전하지 않다 — 마이그레이션 적용 이후 신규로 쓰인 데이터까지 함께 영향을 받을 수 있기 때문이다. 이 경우 §5(Backup/Restore)의 시점 복구(PITR)를 함께 검토한다.

### 4.3 append-only 원장이 있는 상태에서의 롤백 주의사항 (확정 원칙 인용)

- [DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §1.2는 "**append-only 원장 테이블(`commission_records`, `settlement_items`, `audit_logs`): 기존 행을 `UPDATE`/`DELETE`로 직접 수정하지 않는다. 정정은 반드시 보정(역분개) 엔트리 추가로 처리한다**"고 명시한다. 롤백 상황에서도 이 원칙은 예외 없이 적용된다.
- 즉, 배포/마이그레이션 문제로 인해 `commission_records`/`settlement_items`에 잘못된 값이 기록되었더라도:
  - 해당 행을 직접 `UPDATE`/`DELETE`하는 방식의 "롤백"은 금지된다.
  - 대신 보정(역분개) 엔트리를 추가하는 방식으로 정정한다 — [ARCHITECTURE.md](ARCHITECTURE.md) §7이 기술하는 "변경 시점 이후의 계산에만 적용, 과거 확정된 원장은 재작성하지 않는다"는 원칙과 동일한 패턴이다.
- 또한 [DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §1.3은 "worker가 아닌 서비스(web/api/scheduler)에서 핵심 append-only 원장에 직접 쓰지 않는다"고 명시한다 — 롤백 작업이라 해도 worker 경로를 거치지 않고 web/api/scheduler 또는 DB 콘솔에서 이 원장에 직접 쓰는 것은 금지된다.
- 코드 롤백으로 인해 worker의 계산 로직이 이전 버전으로 되돌아갔는데, 그 사이(잘못된 버전이 떠 있던 동안) 이미 생성된 `commission_records`/`settlement_items` 행이 있다면, 그 행들을 일괄 삭제하지 않고 영향받은 건을 식별해 보정 엔트리로 정정하는 절차를 별도로 수행해야 한다 — 이 식별·보정 절차는 금액 영향이 크므로 운영자 승인을 거치는 것을 권장한다.

## 5. Backup / Restore

> **O-037(Supabase 백업 주기/보존기간 및 별도 백업 정책 필요 여부)과 O-144(Supabase 자동백업/PITR 외 RTO/RPO 목표값·리전 장애 DR 절차 정의 여부)는 모두 미확정이다.** 아래는 최소 권장값을 제시하는 것이며, 확정값이 아니다.

### 5.1 현황

- [ARCHITECTURE.md](ARCHITECTURE.md) §2.6에 따르면, 현재 FNS는 **Supabase의 자동 백업/PITR(Point-in-Time Recovery) 기능을 1차(유일한) 백업 수단으로 의존**하고 있다. 별도의 자체 백업 파이프라인은 아직 설계되어 있지 않다.
- 백업 주기·보존기간·별도 백업 정책 필요 여부는 O-037에서, RTO/RPO 목표값과 리전 장애 대응 DR 절차는 O-144에서 각각 추적 중이며 둘은 서로 연계된 항목이다([DECISIONS.md](DECISIONS.md) O-144 비고: "O-037 연계").

### 5.2 최소 권장값 (권장안 — 확정 아님)

| 항목 | 권장 최소값 | 비고 |
|---|---|---|
| RPO(Recovery Point Objective) | 24시간 이내 | Supabase PITR이 제공하는 범위 내에서 설정 — 실제 PITR 보존기간/세분도는 Supabase 플랜 및 O-037 확정 후 결정 |
| RTO(Recovery Time Objective) | 명확한 목표값 미정 — 우선 "당일 내 복구"를 잠정 기준으로 권장 | O-144에서 리전 장애 시나리오까지 포함해 정식 목표값을 확정해야 함 |
| 백업 보존기간 | 최소 7일(일별), 가능하면 30일까지 점진 확대 | Supabase 플랜별 PITR 보존기간 제약을 함께 확인 필요(O-037) |
| 별도 백업 파이프라인 필요 여부 | 현재는 Supabase 의존(별도 파이프라인 없음) — Production 데이터가 커지기 전에 재검토 권장 | O-037에서 정식 결정 |
| DR(리전 장애 대응) 절차 | 현재 정의되어 있지 않음 — 최소한 "정기 백업 export를 별도 리전/스토리지에 보관"을 1차 권장안으로 검토 | O-144에서 정식 결정 |

- 위 값은 어디까지나 **업계 일반적인 최소 기준을 참고한 권장안**이며, FNS의 실제 SLA·법적 요구사항(공제조합 보고 의무 등)을 고려한 최종 목표값은 O-037/O-144가 확정된 후 본 문서를 갱신해 반영한다.

### 5.3 복구(Restore) 절차 (권장안)

1. 복구가 필요한 사고 범위를 먼저 식별한다(전체 DB vs 특정 테이블/기간).
2. Supabase PITR을 이용해 사고 발생 이전 시점으로 복구 가능한지 확인한다.
3. **Production DB를 직접 복구 대상으로 즉시 덮어쓰지 않는다** — 가능하면 별도 인스턴스/스냅샷으로 먼저 복구해 데이터를 검증한 뒤, 검증된 결과를 반영하는 절차를 권장한다.
4. append-only 원장(`commission_records`, `settlement_items` 등)이 관련된 사고라면, 복구 후에도 §4.3의 원칙(직접 수정 금지, 보정 엔트리 우선)을 그대로 적용한다 — "복구"가 "과거 원장을 임의로 재작성"하는 수단이 되어서는 안 된다.
5. 복구 작업 전후 내용을 Audit Log 또는 별도 운영 기록으로 남긴다.

## 6. Health Check

5개 서비스([ARCHITECTURE.md](ARCHITECTURE.md) §1~§2) 각각에 대해 헬스체크 엔드포인트(또는 점검 항목)를 권장한다. 구체적인 경로/구현은 구현 단계에서 확정한다.

| 서비스 | 헬스체크 권장 항목 |
|---|---|
| web | 기본 liveness(프로세스 응답 여부) + `api` 연결 가능 여부(주요 화면이 api 호출에 실패하지 않는지) |
| api | liveness(프로세스 응답) + readiness(Supabase PostgreSQL 연결 가능 여부, Redis 연결 가능 여부) — api가 Job을 생성하지 못하면 전체 비동기 흐름이 막히므로 Redis 연결 상태를 반드시 포함 |
| worker | liveness + Redis 큐 연결 가능 여부 + (가능하면) 큐 적체 길이(pending Job 수)를 함께 노출 — 적체가 비정상적으로 늘어나면 worker 처리 지연을 의심 |
| scheduler | liveness + "정상적으로 1개 인스턴스만 떠 있는가" 점검 — 2개 이상이 동시에 살아있으면 Job 중복 생성 위험([ARCHITECTURE.md](ARCHITECTURE.md) §9.1) |
| redis | liveness(ping) + 메모리 사용량/연결 가능 여부 |

### 6.1 Redis 장애 감지 (O-161 참조)

- [DECISIONS.md](DECISIONS.md) O-161은 "Redis 장애 시 BullMQ Job 손실 감지 및 Dead Letter Queue 재처리 정책"을 **미확정**으로 추적하고 있다 — 아래는 O-161이 확정되기 전까지의 최소 권장 모니터링 범위다.
- Redis는 source of truth가 아니므로([ARCHITECTURE.md](ARCHITECTURE.md) §2.5, [DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §1.3) Redis 장애 자체가 데이터 손실로 직결되지는 않지만, **장애 시점에 처리 중이거나 대기 중이던 Job(특히 정산/수당 계산 Job)이 손실될 위험**은 있으므로 다음을 health check/모니터링에 포함하는 것을 권장한다.
  - Redis 연결 실패를 api/worker/scheduler 각각에서 즉시 감지하고 알림으로 연결한다(§7.2).
  - Redis 장애 복구 후, 장애 구간에 생성되었어야 할 Job(예: scheduler가 트리거하려 했던 정산 배치)이 실제로 적재되었는지 확인하는 점검 절차를 권장한다 — 구체적인 감지/재처리(Dead Letter Queue 등) 메커니즘은 O-161 확정 후 본 문서에 반영한다.

## 7. Monitoring

> **모니터링/로깅 도구 선정(O-025, [ARCHITECTURE.md](ARCHITECTURE.md) §6)은 미확정이다.** 현재는 1차로 Railway 기본 로그를 활용하는 것으로만 정해져 있고, APM/에러트래킹(Sentry 등) 도입 여부는 구현 단계에서 결정한다([ARCHITECTURE.md](ARCHITECTURE.md) §6). 아래는 도구를 특정하지 않고 "무엇을 모니터링해야 하는가"를 권장안으로 정리한다.

### 7.1 모니터링 대상 (권장안)

- 서비스별 가용성/응답시간(§6 Health Check 결과를 시계열로 추적).
- `redis` 큐 적체 길이, 처리 지연 시간.
- Supabase PostgreSQL 연결 풀 사용량, 슬로우 쿼리.
- 5개 서비스의 에러율(특히 api의 5xx, worker의 Job 실패율).
- [ARCHITECTURE.md](ARCHITECTURE.md) §8.1 법적 한도(35%) 모니터링 엔진의 자체 결과(Redis 실시간 카운터, `compliance_ratio_snapshots`)도 인프라 모니터링과 별개로 비즈니스 모니터링 대상에 포함.

### 7.2 금전적으로 중요한 Job의 모니터링 우선순위 (권장안)

[ARCHITECTURE.md](ARCHITECTURE.md) §2.3이 정의하는 worker 처리 대상 중, 실패 시 회원에게 금전적 영향을 주거나 법적 한도 위반으로 이어질 수 있는 Job을 **최우선 모니터링 대상**으로 권장한다.

| 우선순위 | Job 종류 | 실패 시 영향 | 권장 알림 수준 |
|---|---|---|---|
| 최우선 | 정산 배치(Settlement), 법적 한도(35%) 검증 | 회원 지급 누락/오류, 법적 한도 초과 미차단 — 법적·금전적 리스크 직결 | 즉시 알림(실패 즉시 운영 책임자에게 통지), 재시도 소진 시 별도 에스컬레이션 |
| 최우선 | 수당(Compensation) 계산 배치 | 후원수당 산정 오류·지연 | 즉시 알림 |
| 높음 | 조직이동 적용 배치(§7.1.1) | `sponsor_id` 갱신 누락 시 이후 수당/정산 계산이 잘못된 조직 구조를 기준으로 수행됨 | 실패 시 즉시 알림 |
| 높음 | 세금(원천징수) 계산, 지급조서 산출 | 세무 신고 오류 | 실패 시 당일 내 알림 |
| 중간 | 공제조합 보고서 생성(§8) | 규제 보고 지연 | 마감일 임박 + 실패 시 알림 |
| 중간 | 정기배송 자동결제/주문 생성 | 회원 정기배송 누락 | 실패 건수 누적 시 알림 |
| 낮음 | 알림 발송(Email/SMS/KakaoTalk/Push) | 사용자 경험 저하(재시도로 대부분 회복) | 실패율이 임계치를 넘을 때만 알림 |
| 낮음 | 3PL 정합성 대조, 통계/리포트 집계 | 운영 편의성 저하, 즉각적 금전 영향 적음 | 일/주 단위 요약 보고 |

- "최우선"/"높음" 등급 Job은 실패 시 **재시도(Redis Retry)만으로 종료 처리하지 않고**, 재시도가 모두 소진되었을 때 사람이 즉시 인지할 수 있는 알림 경로(운영 책임자 대상)를 갖추는 것을 권장한다 — 구체적인 알림 채널/도구는 O-025 확정 후 결정한다.
- 법적 한도(35%) 모니터링은 [ARCHITECTURE.md](ARCHITECTURE.md) §8.1의 3종 계산 단위(실시간/월별/연도누적) 각각의 정합화 배치가 정상 수행되는지를 별도로 모니터링한다 — 실시간 캐시(Redis)와 Postgres 간 정합화 주기 자체도 미확정(O-078)이므로, 정합화가 누락되는 상황을 감지하는 것이 1차 목표다.

## 8. 후속 갱신 예정

- O-022(ORM/마이그레이션 도구), O-025(모니터링/로깅 도구), O-037(백업 정책), O-144(RTO/RPO·DR), O-148(환경 분리)이 각각 확정되는 대로, 본 문서의 해당 섹션을 "권장안"에서 "확정 절차"로 갱신한다.
- O-161(Redis 장애 시 DLQ 재처리 정책), O-070(조직이동 적용 배치 주기), O-078(법적 한도 실시간/Postgres 정합화 주기)이 확정되면 §6.1/§7.2의 관련 서술도 함께 갱신한다.
