### D-010. 서버 구조를 Railway 프로젝트 1개 + 5개 서비스로 확정한다
- **날짜**: 2026-06-23
- **결정**: FNS의 최종 서버 구조는 Railway 프로젝트 1개 안의 **web(Next.js) / api(NestJS) / worker(NestJS) / scheduler(NestJS) / redis** 5개 서비스로 확정한다. Supabase는 PostgreSQL/Auth/Storage/Backup을 담당하는 외부 관리형 서비스로 둔다.
- **핵심 원칙**:
  - web에서 수당 등 계산 금지 (UI만)
  - api의 request handler에서 무거운 계산 금지 — **api는 Job 생성만 담당**
  - **worker가 Job을 실제 처리** (수당/정산/세금/프로모션/보고서)
  - **scheduler는 Job 트리거만 담당** (자동정산/자동프로모션/자동보고서), 직접 계산하지 않음
  - **Redis는 source of truth가 아님** — 최종 데이터는 **Supabase PostgreSQL에만 저장**
- **근거**: 사용자(프로젝트 오너) 확정 지시 — 요청 처리와 무거운 계산을 물리적으로 분리하여, 정산/수당 배치처리량이 늘어나도 web/api 응답성에 영향을 주지 않도록 함.
- **관련 문서**: [ARCHITECTURE.md](ARCHITECTURE.md) §1~§2/§8, [PROJECT-CONTEXT.md](PROJECT-CONTEXT.md) §6.1, [TASK-SPEC.md](TASK-SPEC.md) §6, [DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §1.3

### D-013. Database Review / Architecture Review 결과를 기록한다 (Design Freeze 검토)
- **날짜**: 2026-06-23
- **결정**: 현재 데이터 모델과 Railway 5서비스 구조에 대해 누락 테이블/컬럼, 성능·확장성·법률·운영 위험을 점검했다 ([DATABASE.md](DATABASE.md) §7, [ARCHITECTURE.md](ARCHITECTURE.md) §9). 가장 중대한 발견:
  1. **CN(중국) 회원 데이터의 현지화 법규 충돌 가능성** — 단일 Supabase 리전 사용이 PIPL 등과 충돌할 수 있어 CN 출시 전 법률 검토가 필수다.
  2. **closure table(`member_ancestors`) 도입을 구현 1순위로 격상** — LINE1~5 조회가 누적 데이터 영역 확대로 더 빈번해짐.
  3. **Job 재처리 멱등성 부재** — 수당/알림 등 worker Job이 재시도 시 중복 실행될 위험이 있어 `idempotency_key` 도입이 필요하다.
  4. **단일 worker/redis가 처리하는 Job·트래픽 종류가 5종→9종으로 증가** — 계산형/IO형 분리, 캐시/큐 Redis 분리를 확장 1순위 후보로 권고한다.
- **근거**: 사용자 요청에 따른 Design Freeze 시점의 자체 검토(Self-audit). 즉시 구현을 요구하는 결정은 아니며, 우선순위가 높은 위험을 식별해 다음 단계(구현 또는 추가 설계)의 입력으로 남긴다.
- **관련 문서**: [DATABASE.md](DATABASE.md) §7, [ARCHITECTURE.md](ARCHITECTURE.md) §9, 본 문서 §3 "Design Freeze Report"

### D-020. 추천인 변경을 회원 자기서비스에서 관리자 전용 "조직 이동"으로 전환한다 (D-017 대체, O-024 해소)
- **날짜**: 2026-06-23
- **결정**:
  1. **회원은 추천인(스폰서)을 스스로 변경할 수 없다.** 회원 화면에는 "추천인 변경", "조직 이동", "변경 신청" 기능을 제공하지 않는다 — 회원은 조회만 가능하다.
  2. 추천인 변경은 **관리자 전용 기능**으로 전환하며, 통일 용어는 **"조직 이동"** (= 추천인 변경)으로 한다. 이 용어는 **관리자 화면에서만 사용**하며 회원 화면에는 노출하지 않는다.
  3. 조직 이동은 다음 **8개 사유에 한해서만** 가능하다: 회원 탈퇴, 회원 사망, 명의 변경, 운영 오류 수정, 회사 정책상 조직 재배치, 법원 판결, 법적 조치, 기타 회사 승인 사유.
  4. 모든 조직 이동은 다음 **7단계 절차**를 따른다: ① 변경 사유 입력 → ② 관련 증빙 첨부 → ③ 영향 분석 → ④ Snapshot 생성 → ⑤ 관리자 승인 → ⑥ 적용 → ⑦ Audit Log 저장.
  5. 조직 이동은 **과거 수당·정산·직급을 수정하지 않으며**, 변경 시점 이후의 계산에만 반영한다 (기존 원칙과 동일, 재확인).
  6. 조직 이동 권한은 **SuperAdmin, Compliance Admin, 지정 조직관리 관리자**에게만 부여하며, 일반 관리자에게는 부여하지 않는다.
  7. 관리자 메뉴 "조직관리"에 조직 조회/조직 이동/조직 이동 이력/조직 이동 영향분석/조직 병합/조직 분리/조직 복구/조직 통계 8개 항목을 둔다. **조직 병합/분리/복구는 이번 결정에서 메뉴 구조만 확정하며, 기능 정의는 미확정**(신규 Open Decision)이다.
  8. 조직 이동 이력 조회, 전후 비교, 영향 분석, 승인 이력, 첨부파일 관리, 사유 코드 관리 기능을 **필수**로 추가한다.
- **근거**: 사용자(프로젝트 오너) 확정 지시. D-017(전면 허용)이 안고 있던 어뷰징·금지행위 저촉 위험(O-024)을 "회원이 아예 신청할 수 없고, 회사가 제한된 사유로만 수행"하는 구조로 바꿔 근본적으로 제거하는 방향.
- **O-024 처리**: 정책이 "전면 허용"에서 "회원 신청 불가·관리자 전용·제한된 8개 사유"로 바뀌어 어뷰징(조직 강탈) 위험의 전제 자체가 사라졌으므로 **O-024는 해소된 것으로 처리**한다. 다만 "기타 회사 승인 사유"라는 포괄 조항이 남아있어, 이 조항의 오남용 방지 장치(승인권자 제한, 사유 기록 의무)가 충분한지에 대한 가벼운 법률 확인은 신규 Open Decision(O-064)으로 남긴다.
- **관련 문서**: [PRD.md](PRD.md) §5.3/§5.9(신규), [ARCHITECTURE.md](ARCHITECTURE.md) §2.2/§7, [DATABASE.md](DATABASE.md) §3.9/§3.26(신규), [LEGAL-CHECKLIST.md](LEGAL-CHECKLIST.md) §10

### D-021. 회원 탈퇴 시 조직 구조는 자동 변경하지 않는다 (D-020의 직접적 귀결, 누락 보완)
- **날짜**: 2026-06-23
- **결정**:
  1. 회원 탈퇴 시 **조직 구조(추천관계)는 자동으로 변경하지 않는다.** 탈퇴 회원은 `members.status`가 **WITHDRAWN**으로 전환되지만, **기존 `sponsor_id` 관계와 하위 조직 구조는 그대로 유지**한다.
  2. WITHDRAWN 회원은 **신규 후원수당·직급(승급/유지/강급)·정산·프로모션 대상에서 제외**된다 — 탈퇴 시점 이후 발생하는 모든 계산 배치에서 수령자/판정 대상으로 포함하지 않는다.
  3. 단, WITHDRAWN 회원은 **추천조직 트리 내 노드로서는 그대로 남는다** — 다른(상위) 회원의 LINE1~5 도달 깊이 계산은 이 노드를 그대로 통과하며 영향받지 않는다(조직 구조 자체를 바꾸지 않으므로). 즉 "수령 자격 제외"와 "트리 구조 보존"은 별개로 처리한다.
  4. 조직 재배치가 필요한 경우(예: 탈퇴한 사람의 하위 조직을 다른 추천인 밑으로 옮기고 싶을 때)에는 **관리자 전용 조직 이동(§3.26, D-020) 기능을 통해 별도로, 수동으로 처리한다.** "회원 탈퇴"는 조직 이동의 8개 허용 사유 중 하나이지만(D-020 §5.16.3), **탈퇴 승인이 조직 이동을 자동으로 트리거하지는 않는다.**
  5. **과거 수당·정산·직급·조직 이력은 그대로 유지한다** (기존 append-only 원칙의 재확인 — 탈퇴 자체가 과거 기록에 어떤 영향도 주지 않음).
- **근거**: 사용자(프로젝트 오너) 확정 지시. 직전 대화에서 "탈퇴 시 하위 조직이 어떻게 되는가"가 설계에 명시되어 있지 않다는 점이 발견되어, 이를 공식 정책으로 확정했다.
- **관련 문서**: [PRD.md](PRD.md) §5.3, [COMPENSATION-RULES.md](COMPENSATION-RULES.md) §3.3/§5, [PROMOTION-RULES.md](PROMOTION-RULES.md) §6, [SETTLEMENT-RULES.md](SETTLEMENT-RULES.md) §9, [DATABASE.md](DATABASE.md) §3.1/§3.12, [ARCHITECTURE.md](ARCHITECTURE.md) §7

### D-022. 조직 이동의 "승인"과 "적용"을 분리한다 — 적용일(effective_date) 예약 메커니즘 도입 (D-020/D-021 보강)
- **날짜**: 2026-06-23
- **결정**:
  1. **조직 이동 승인 ≠ 조직 이동 적용.** 승인은 "이 변경을 허락한다"는 의사결정이고, 적용은 "실제로 `sponsor_id`를 바꾸는 시점"이며 둘은 서로 다른 시각에 일어난다. 모든 `organization_transfer_logs` 행은 **승인일(approved_at)과 적용일(effective_date)을 별도로** 가진다.
  2. **기본 정책**: 승인 후 **익월 1일 00:00**에 적용한다 (예: 신청 2026-06-15, 승인 2026-06-20 → 적용 2026-07-01 00:00). 기준 시점은 **승인일**로 해석한다(승인일이 월말 근처라 신청일 기준으로 계산하면 이미 지나버린 날짜가 나올 수 있는 edge case를 피하기 위함 — 원문에 명시되지 않아 확정 필요, [DECISIONS.md](DECISIONS.md) Open Decisions 참조).
  3. **적용 전까지는 기존 조직 구조·수당·직급·정산을 그대로 유지**한다. 승인되었다는 사실만으로 `members.sponsor_id`나 계산 결과가 바뀌지 않는다.
  4. **적용일 이후부터** 신규 조직 구조를 기준으로 수당·직급·정산을 계산한다.
  5. **과거 수당·정산·직급·조직 이력은 변경하지 않는다** (append-only 원칙 재확인 — D-004/D-021과 동일).
  6. **긴급 조직 이동**: 다음 5개 사유에 한해서만 승인 즉시 적용(적용일 예약 절차를 생략)할 수 있다 — 회원 사망, 법원 판결, 법적 조치, 시스템 운영 오류, 중대한 컴플라이언스 이슈. 이 중 "중대한 컴플라이언스 이슈"는 기존 8개 사유([DECISIONS.md](DECISIONS.md) D-020)에 없던 **신규 사유로 추가**한다(9번째 사유 코드).
  7. 긴급 조직 이동은 **SuperAdmin만 승인 가능**하다 — Compliance Admin/지정 조직관리 관리자는 일반(예약형) 조직 이동만 승인할 수 있고 긴급은 승인할 수 없다(D-020의 3개 승인 가능 역할보다 더 좁힌 것). **Audit Log와 변경 사유 입력은 긴급 여부와 무관하게 항상 필수**였으나, 긴급의 경우 누가 "긴급"으로 분류했는지도 함께 기록한다.
  8. `organization_transfer_logs`에 **`effective_date`, `is_emergency`** 컬럼을 추가한다 (`requested_at`/`approved_at`/`applied_at`은 기존에 이미 있음).
  9. 처리 절차를 7단계에서 **9단계**로 갱신한다: ① 조직 이동 요청 → ② 증빙 첨부 → ③ 영향 분석 → ④ Snapshot 생성 → ⑤ 승인 → ⑥ 적용일 예약 → ⑦ 적용일 도달 → ⑧ 실제 조직 이동 적용 → ⑨ Audit Log 저장. 긴급 조직 이동은 ⑥~⑦을 생략하고 ⑤ 승인 직후 곧바로 ⑧로 진행한다(③ 영향 분석은 긴급이라도 생략하지 않는다 — 가능한 한 신속히 수행).
  10. **수당/직급/정산 엔진 공통 원칙**: 계산은 항상 계산일 기준으로 **이미 적용(applied)된 조직 구조만** 사용한다. 승인되었지만 아직 적용일이 도달하지 않은 조직 이동은 어떤 계산에도 영향을 주지 않는다. 실무적으로 이는 "`members.sponsor_id`는 적용 시점에만 갱신된다"는 원칙(②)을 그대로 따르면 자동으로 보장된다 — 계산 엔진이 별도로 "승인 상태"를 확인할 필요는 없다.
- **근거**: 사용자(프로젝트 오너) 확정 지시 — "조직 이동은 수당/직급/정산에 직접 영향을 주므로 승인과 적용을 반드시 분리해야 한다."

### D-023. 지원 국가를 4개국(KR/TH/JP/US)으로 축소하고, CN(중국)을 "Reserved Country"로 전환한다 (D-011 부분 수정)
- **날짜**: 2026-06-23
- **결정**:
  1. **지원(활성) 국가는 KR / TH / JP / US 4개국으로 확정**한다. CN은 1차 및 중기 계획에서 **제외**한다.

  5. CN을 다시 활성 국가로 전환(Reserved 해제)하려면 별도의 명시적 의사결정(신규 D-번호)이 필요하다 — 자동으로 복원되지 않는다.
- **근거**: 사용자(프로젝트 오너) 확정 지시 — 1차 및 중기 계획에서 중국을 제외하기로 함. 데이터/문서를 삭제하지 않고 보관하는 이유는 향후 확장 옵션을 유지하기 위함.
- **이전 결정과의 관계**: [D-011](#1-확정된-결정-decided)의 "지원 국가 5개국" 부분을 수정한다 — 회원 유형(4종) 부분은 변경 없음. [D-013](#1-확정된-결정-decided)/O-053의 CN 법률 리스크 자체는 폐기하지 않고 우선순위만 낮춘다(Reserved 해제 시 재상향).
- **관련 문서**: [PROJECT-CONTEXT.md](PROJECT-CONTEXT.md) §3/§4/§10, [PRD.md](PRD.md) §5.6.2, [ARCHITECTURE.md](ARCHITECTURE.md) §8, [DATABASE.md](DATABASE.md) §3.13/§7.5, [LEGAL-CHECKLIST.md](LEGAL-CHECKLIST.md)(신규 항목)

### D-026. 페어보너스(Pair Bonus) 실패 규칙을 최종 확정한다 (O-072 해소, D-024 §6 보강)

- **날짜**: 2026-06-24
- **결정**: D-024에서 도입한 페어보너스(§4.1.3)의 미해결 큐 재정렬 규칙(O-072)을 다음과 같이 확정한다.
  1. 추천인별로 "짝을 기다리는 대기 후보"는 최대 1건만 존재한다.
  2. 대기 후보의 **30일 Pair Window가 새 구매자 없이 만료되면, 그 Pair Bonus는 소멸하고 해당 구매는 종료(closed) 처리**된다.
  3. **재사용 금지·자동 이월 없음·자동 재매칭 없음** — 만료된 구매는 이후 어떤 Pair 계산에도 다시 사용되지 않으며, 다음에 들어오는 구매자와 자동으로 묶이지 않는다.
  4. 예시: A 구매 → 30일간 짝 없음 → A 종료(소멸). 이후 B·C가 (30일 이내로) 구매 → **B+C가 Pair**(200만원) 성립, **A는 포함되지 않는다.**
- **근거**: 사용자(프로젝트 오너) 확정 지시 — D-024 당시 "원문에 명시되지 않음"으로 보류했던 부분을 명확화.
- **O-072 해소**: Open Decisions 표에서 제거한다.

#### O-059 영향분석 — 분류 시나리오별 35% 계산식 변화 (요청에 따른 시나리오 분석)

패키지 추천보너스(25%)·페어보너스(Pair당 200만원)의 법적 분류(O-059)에 따라 35% 분자(分子)의 구성이 달라진다. p = 동일 추천인 산하 패키지 구매자의 **페어 성립률**(0~1)이라 하면:

- **패키지 매출만 따로 본 커미션 비율 = 25% × (1 + p)** — 추천보너스(25%, 항상 발생)에 페어보너스(구매 2건당 200만원=8,000,000원의 25%, 단 p의 비율만큼만 발생)가 더해지는 구조다.
  - p=0(페어 전혀 안 됨): 25%
  - **p=0.4(40% 페어 성립): 35% — 패키지 매출만 떼어봐도 이미 법정 한도에 도달**
  - p=1(전부 페어 성립): 50%
- **시나리오 A — 둘 다 "후원수당"으로 분류**: 분자 = 유니레벨 후원수당 + 패키지 추천보너스 + 페어보너스(+기타 채택 수당). 분모 = 전체 매출(일반제품+패키지). 패키지 매출 비중이 커지고 페어 성립률이 높아질수록(p≥40%) **패키지 관련 커미션만으로 법정 한도에 근접·초과할 수 있다** — 회사가 패키지 판매를 적극 확장하는 성장 시나리오에서 실질적 위반 위험이 존재한다.
- **시나리오 B — 둘 다 "제품 판매이익"으로 분류**: 분자 = 유니레벨 후원수당(+기타 채택 수당)만 — 패키지 추천보너스/페어보너스는 분자에서 완전히 빠진다(매출 자체는 분모에는 그대로 포함). 이 경우 패키지 판매량이 늘어나도 35% 한도 위반 위험이 구조적으로 거의 사라진다.
- **시나리오 C — 혼합 분류**(예: 패키지 추천보너스만 판매이익, 페어보너스는 조직실적 조건부 가산이라 후원수당): 분자 = 유니레벨 후원수당 + 페어보너스만. A와 B의 중간 위험도.
- **현재 임시 기본값(D-024/§6)은 시나리오 A**(보수적) — 즉 현재 모니터링 엔진은 가장 위험도가 높은 가정으로 동작 중이다. O-059가 시나리오 B로 최종 확정되면 한도 위반 위험이 크게 줄어든다.
- **권고**: O-059 최종 확정 전까지는 ① 시나리오 A 기준으로 모니터링을 유지하고, ② **페어 성립률(p)과 패키지 매출 비중을 별도 리딩 인디케이터로 대시보드에 노출**해 블렌딩된 전체 비율이 30%를 넘기 전에 조기 경보를 받을 수 있게 한다([ARCHITECTURE.md](ARCHITECTURE.md) §8.1.5, [PRD.md](PRD.md) §5.18).
- **관련 문서**: [COMPENSATION-RULES.md](COMPENSATION-RULES.md) §4.1/§6, [ARCHITECTURE.md](ARCHITECTURE.md) §8.1, [DATABASE.md](DATABASE.md) §3.29, [PRD.md](PRD.md) §5.18, [LEGAL-CHECKLIST.md](LEGAL-CHECKLIST.md) §3/§9

### D-028. 유니레벨 후원수당 자격과 "제품 판매수익·페어보너스" 자격을 분리하고, "패키지 추천보너스"를 "제품 판매수익"으로 개칭한다 (D-024 §2/§5/§6 정정)

- **날짜**: 2026-06-24
- **결정**: D-024가 "5만원 일반제품 구매 또는 400만원 패키지 구매" 둘 중 하나를 충족하면 §4 전체 수당에 대한 "후원수당 자격"을 준다고 정의한 것은 **잘못되었다.** 자격을 다음 2개로 완전히 분리한다.
  1. **유니레벨 후원수당 자격**: 매월 5만원 상당 제품 구매로 유지한다. 무료회원가입 후 5만원 상당 제품을 구매하면 그 달부터 유니레벨 후원수당을 받을 수 있다. 매월 구매가 없으면 그 달의 유니레벨 후원수당을 받을 수 없다([COMPENSATION-RULES.md](COMPENSATION-RULES.md) §3.5.2).
  2. **제품 판매수익·페어보너스 자격**: 제품 판매수익(100만원)과 페어보너스(200만원)를 받으려면 **본인이 먼저 400만원 패키지를 구매한 적이 있어야 한다.** 본인이 패키지를 구매하지 않았다면, 직추천 회원이 패키지를 구매하거나(제품 판매수익 조건) 직추천 회원 2명이 30일 이내에 패키지를 구매해(페어보너스 조건) 그 조건이 충족되어도 **지급하지 않는다.** 지급되지 않은 수당은 누구에게도 이전되지 않고 소멸한다(pass-up 없음) ([COMPENSATION-RULES.md](COMPENSATION-RULES.md) §3.5.5).
  3. **예시**: "나"는 무료회원가입 후 5만원 제품을 구매했다. 내 직추천 A는 회원가입 후 400만원 패키지를 구매했다. → 나는 유니레벨 후원수당 자격은 있다. 그러나 나는 400만원 패키지를 구매하지 않았으므로 제품 판매수익 100만원은 받을 수 없다. 페어보너스 조건이 충족되더라도 페어보너스도 받을 수 없다.
  4. **명칭 정정**: "패키지 추천보너스(Package Referral Bonus)"라는 명칭을 "**제품 판매수익(Package Sales Profit / 패키지 판매수익)**"으로 바꾼다 — 실제 의미가 "추천 활동에 대한 보상"보다 "본인이 자격(=본인 구매)을 갖춘 상태에서 발생하는 판매 실적에 대한 수익"에 더 가깝다고 판단했기 때문이다. 수령자·트리거 이벤트·금액(25%/100만원, Pair당 200만원)은 변경되지 않는다.
- **근거**: 사용자(프로젝트 오너) 확정 지시 — D-024 당시 두 자격을 하나로 묶은 것이 원본 의도와 다르다는 점을 재검토 후 정정.
- **D-024와의 관계**: D-024 §2(자격 최초충족 OR게이트), §5(패키지 추천보너스 명칭), §6(페어보너스 수령자 자격 미명시)을 본 결정으로 **정정**한다. 나머지 항목(LINE 비율, 산정매출기준, 페어링 순서규칙, 30일 윈도우, 4축 구조)은 변경 없다.
- **D-027과의 관계**: D-027(같은 날, 본 결정 이전에 작성)의 O-059 시나리오 분석에서 사용된 "패키지 추천보너스"라는 표현은 본 결정 이후로는 "제품 판매수익"으로 읽는다 — 분석 내용(35% 계산식) 자체는 변경되지 않는다.
- **법적 분류(O-059)에 대한 추가 시사점**: "제품 판매수익"이라는 명칭과 본인 구매 선행조건은 "제품 판매이익"(자기 자격 보유자의 판매 마진) 해석에 무게를 더할 수 있으나, 지급 사유가 여전히 "직추천 회원의 구매 실적"이라는 점에서 후원수당적 성격도 동시에 존재한다 — 두 요소가 서로 다른 방향을 가리키므로 O-059는 여전히 Open Decision으로 남긴다([COMPENSATION-RULES.md](COMPENSATION-RULES.md) §4.1).
- **신규 법적 리스크 식별 — O-081**: 직추천 회원을 둔 회원이 제품 판매수익·페어보너스를 받으려면 거의 필연적으로 본인도 400만원 패키지를 구매해야 하는 경제적 압박이 생긴다 — 이는 가입 자체를 강제하지는 않지만, **"보너스를 받으려면 사실상 고액 구매가 필요한 구조"** 가 방문판매법상 금지행위(사재기 강요 등)와 어떤 관계에 있는지 법률 검토가 필요하다. [LEGAL-CHECKLIST.md](LEGAL-CHECKLIST.md) §5, [DECISIONS.md](DECISIONS.md) §2 O-081로 추적.

### D-029. 정산 주기를 월정산 단일로 정정한다 (D-016 정정, 정합성 전면 점검)

- **날짜**: 2026-06-24
- **결정**: FNS는 **월정산 1회만 사용**한다. 주정산은 사용하지 않는다.

- **연쇄 영향**: D-016의 "직급수당은 월정산" 부분은, 직급수당 자체가 현재 FNS 마케팅 플랜에 채택되지 않은 개념이라는 점([COMPENSATION-RULES.md](COMPENSATION-RULES.md) §4, 직급수당 행은 항상 "미확정"이었음)과 맞물려, 본 결정의 "정합성 전면 점검" 라운드에서 §5.10 보고서로 함께 다룬다 — 직급 체계 자체의 존속 여부는 별도 분석 대상이다(D-029는 정산 주기만 확정, 직급 구조 자체의 폐기 여부는 미확정으로 남김).

### D-030. 직급(Rank) 체계와 PV(Point Value) 추상화 계층을 폐기한다 (D-015 폐기, O-083/O-084 해소, 정합성 점검 §5.10 실행)

### D-031. 마케팅 플랜 재정의 최종 점검 — 잔여 정합성 이슈 해소, 쇼핑몰(정기배송) 구조 신설 (사용자 요청 "FNS 마케팅 플랜 재정의" 라운드)

- **날짜**: 2026-06-24
- **결정**: 사용자가 "기존 MLM 일반 구조를 배제하고 FNS 실제 사업 모델만 기준으로 재정리"를 요청한 라운드에서, D-030 실행 이후에도 남아있던 잔여 정합성 문제를 해소하고, 기존 문서에 빠져 있던 쇼핑몰 구조를 신설한다.
  1. **잔여 정합성 정리(문서 충돌 해소)**: D-030이 `pv_ledger`/`member_rank_history`를 폐기했음에도, [DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §1.2(2곳)와 [ARCHITECTURE.md](ARCHITECTURE.md) §2.3가 이 두 테이블을 여전히 "수정 금지 append-only 원장" 목록에 포함하고 있었다 — 폐기된 테이블을 보호 대상으로 잘못 남겨둔 충돌이다. 두 문서에서 해당 테이블명을 제거했다. 추가로 [ARCHITECTURE.md](ARCHITECTURE.md) §2.4/§7, [PRD.md](PRD.md) §5.4가 더 이상 §6을 갖지 않는 [PROMOTION-RULES.md](PROMOTION-RULES.md)(폐기 안내로 대체됨)의 "§6"을 가리키는 깨진 참조를 갖고 있어 D-021/D-022 원칙에 대한 직접 참조로 교체했다. [ARCHITECTURE.md](ARCHITECTURE.md) §2.3의 worker Job 처리 대상 개수도 "9종"으로 남아 있었으나 실제 나열된 항목은 10종(조직이동 적용 Job 포함, D-022)이라 정정했다.
  2. **쇼핑몰 구조 신설**: 기존 PRD/DATABASE 문서에는 "제품/주문" 모듈이 "제품 등록, 주문 생성" 한 줄로만 정의되어 있어, 일반 쇼핑몰/회원몰/정기배송/자동결제/유지구매 관리라는 FNS의 실제 쇼핑몰 구조가 누락되어 있었다. 사용자 지시에 따라 다음을 확정한다.
     - **회원몰**(파트너 자기소비/유지구매 채널)과 **일반 쇼핑몰**(고객 대상, 기존 "고객몰(B2C)"과 동일 개념으로 명칭 일원화)을 구분한다. 일반 쇼핑몰의 Phase 2/포함여부 미확정 상태(O-017)는 변경하지 않는다.
     - **정기배송(Recurring Delivery)을 핵심 기능(MVP)으로 확정**한다 — §3.5.2 유니레벨 자격(매월 5만원 누적 구매)이 매월 능동적 구매에만 의존하면 자격 변동(이탈)이 잦아지므로, 정기배송이 이 자격을 안정시키는 핵심 리텐션 장치이기 때문이다.
     - **자동결제**는 정기배송의 전제조건으로 MVP에 포함하며, PG 토큰화를 전제로 카드 원본 정보는 FNS DB에 저장하지 않는다.
     - **유지구매 관리** 화면(정기배송 등록 현황/당월 결제 이력/유지구매 부족분 안내)을 §5.1.2 자격 현황 화면의 확장으로 추가한다.
     - 정기배송 등록/변경/해지는 조직·수당 구조에 영향을 주지 않으므로 §5.4 민감 변경 승인 워크플로우의 대상이 아니다 — 회원 셀프서비스로 즉시 처리한다.
- **근거**: 사용자(프로젝트 오너) 확정 지시 — "FNS 마케팅 플랜 재정의" 작업의 일부로, 문서 전반의 잔여 충돌을 점검하고 쇼핑몰 구조를 FNS 고유 기준으로 정의하도록 요청.
- **관련 문서**: [PRD.md](PRD.md) §5.1.3(신규), [DATABASE.md](DATABASE.md) §3.30(신규), [DO-NOT-TOUCH.md](DO-NOT-TOUCH.md) §1.2, [ARCHITECTURE.md](ARCHITECTURE.md) §2.3/§2.4/§7

### 5.6 정리

FNS의 설계는 **"어떻게 만들 것인가"(구조/아키텍처/데이터 모델)는 상당히 성숙했고, "무엇을 얼마로 할 것인가"(실제 비율/세율/법규/도입 여부)는 대부분 비어 있다.** 이는 설계 단계에서 자연스러운 상태이며, 구조가 먼저 확정되어야 비즈니스 수치를 안전하게 끼워 넣을 수 있다는 점에서 순서상 올바르다.

**다음 단계 권장 순서 (D-023 반영 갱신)**: (1) KR 기준 LINE 비율/직급체계/정산주기 확정 → (2) KR로 구현 착수(§5.4 영역) → (3) 구현과 병행하여 TH/JP/US 법규·규칙 조사 → (4) 센터/전자서명/Rule Designer 도입 여부를 사업 일정에 맞춰 별도 결정 → (5) **CN 활성화(Reserved 해제)를 검토하게 되면, 그 시점에 CN 법률 리스크(O-053) 확인을 최우선으로 재배치**.

### 5.7 최종 Design Freeze 합의 — Phase 1(KR) 설계 종료 (2026-06-23)

사용자(프로젝트 오너)가 별도로 "FNS 설계 현황 최종 점검 보고서"를 제시하며 종합 완성도를 **95%**로, 설계 종료에 필요한 핵심 결정을 **5개 항목**으로 정리했다. 본 문서 §5.1의 자체 평가(~58%)와 차이가 있어 이를 명시적으로 정리한다.

- **평가 방식의 차이**: §5.1은 "비즈니스 수치·법무 디테일까지 채워졌는가"를 기준으로 했고, 사용자의 95%는 "구조/메커니즘이 설계되었는가"를 기준으로 했다. 두 기준 모두 타당하며, **"KR Phase 1 출시를 막는 항목이 몇 개 남았는가"** 관점에서는 사용자의 5개 항목 분류가 정확하다 — CN 법규(O-053), 센터/전자서명/Rule Designer 도입 여부(O-047/O-051/O-052), 운영 리스크(O-054~O-056) 등은 **KR Phase 1 자체를 막지 않으므로** 5개 항목에서 제외된 것이 합리적이다.
- **5개 항목 중 4개 확정**: LINE 비율(D-014), 직급 체계(D-015, 승급조건은 구조만 확정·임계값은 잔여), 정산 주기(D-016), 추천인 변경 정책(D-017). 사용자는 4개 항목 모두에서 제가 제시한 추천안과 다른 선택을 했다(백로딩/5단계/혼합정산/전면허용) — 추천안은 참고용일 뿐 최종 결정권은 사업팀에 있다는 것을 보여주는 사례로 기록해 둔다.
- **남은 5번째 항목(직급 승급조건)**: 구조(개인PV/그룹PV/직추천수/하위직급보유)는 D-015로 확정했으나, 구체적 PV 임계값은 제품 가격/PV 환산표가 있어야 정할 수 있어 **이번 라운드에서는 미해결로 남긴다** — 사업팀이 제품 가격표를 제공하면 즉시 후속 라운드로 마감 가능하다.
- **결론**: 위 4개 항목 확정 + 5번째 항목의 구조 확정으로, **FNS Phase 1(KR) 설계는 사실상 종료되었다.** 남은 것은 (a) PV 임계값 수치 입력(제품가격표 필요), (b) O-024 법률 자문(추천인 변경 전면허용의 적법성) 두 가지뿐이며, 이 두 가지를 제외한 모든 영역은 [§5.4](#54-즉시-개발-가능한-영역)의 "즉시 개발 가능한 영역"에 더해 **실제 수당/정산/직급 엔진 구현까지 착수 가능**하다 (PV 임계값과 법률 자문 결과는 구현과 병행해 채워도 무방 — 구조는 이미 확정되어 있으므로).

### 5.8 후속 정정 — 회사 보상플랜 원본 문서 반영 (2026-06-23, D-018/D-019)

§5.7 선언 직후, 사용자가 회사의 실제 보상플랜 원본 문서(`COMPENSATIONAL MODEL_V2`)를 제공했다. 검토 결과 두 가지 변경이 발생했다:

1. **D-014의 LINE 산정 방식 자체가 틀렸음을 발견 → D-018로 정정**. 비율 숫자(3/4/5%)는 그대로지만, "라인별 독립 계산 후 합산"이 아니라 "라인의 도달 깊이에 따른 단일 비율을 라인 전체에 적용"하는 방식이다. **이 시점까지 코드를 작성하지 않았기 때문에 구현 오류로 이어지지 않고 설계 단계에서 정정할 수 있었다** — docs-first 워크플로우([PROJECT-CONTEXT.md](PROJECT-CONTEXT.md) §9)의 효과를 보여주는 사례.
2. **신규 수당 2종 발견 → D-019로 추가**: 제품 팩 판매 수익(25%, 페어보너스), "+알파" 보너스(여행/자동차/자기계발).
3. **Phase 1 종료 조건 갱신**: §5.7의 "남은 2가지"에 **O-059(제품 팩 판매 수익의 법적 분류)** 가 O-024와 동급의 최우선 항목으로 추가되어, 이제 **법률 자문이 필요한 항목이 2건(O-024, O-059)** 이 되었다. 나머지 결론(즉시 개발 가능 영역)은 유지된다.

> **교훈**: 사업팀이 보유한 원본 기획/기획서 문서는 설계 초반에 최대한 빨리 공유받는 것이 바람직하다 — 이번에는 운 좋게 코드 작성 전에 발견되었지만, 구현 단계 이후였다면 재작업 비용이 컸을 것이다.

### 5.9 정책 전환 — 추천인 변경을 관리자 전용으로 (2026-06-23, D-020)

§5.8 이후, 사용자가 D-017("추천인 변경 전면 허용")을 재검토하고 **회원 자기서비스를 전면 폐지**하기로 결정했다. 새 정책(D-020)의 핵심:

- 회원은 추천인을 스스로 변경할 수 없다 — 조회만 가능.
- "조직 이동"(=추천인 변경)은 **관리자 전용** 기능이며, 8개의 제한된 사유(탈퇴/사망/명의변경/운영오류/정책재배치/법원판결/법적조치/기타승인)에 한해서만, SuperAdmin/Compliance Admin/지정 조직관리 관리자만 수행 가능.
- 7단계 절차(사유입력→증빙첨부→영향분석→Snapshot→승인→적용→Audit Log)를 의무화.

**O-024(법률 위험)에 대한 효과**: D-017이 안고 있던 핵심 위험은 "회원이 제한 없이 신청 가능"하다는 점이었다. D-020은 이 전제 자체를 제거했으므로(회원은 신청 자체를 할 수 없음), **O-024는 추가 법률 자문 없이 D-020으로 해소된 것으로 처리**한다. 다만 "기타 회사 승인 사유"라는 포괄 조항의 오남용 방지는 경량 법률 확인(O-064)으로 남겨둔다.

**완성도 영향**: 회원 생애주기 관리 영역의 법적 위험이 줄어 §5.1 "법적 컴플라이언스" 점수에 긍정적 영향이 있으나, 조직 병합/분리/복구라는 새 미정의 기능(O-065)이 추가되어 "신규 검토 영역"에는 새로운 미확정 항목이 생겼다 — 순효과는 중립에 가깝다.

- 다국가/컴플라이언스 구조(D-011/D-023)

#### 5.10.4 법률 검토 필요 항목 — O-081 심화 분석

> ⚠️ 아래는 **법률 리스크 신호를 정리한 것이며, 적법/위법에 대한 최종 판단이 아니다.** 최종 판단은 법률 자문으로만 확정한다.

"본인이 먼저 400만원 패키지를 구매해야 제품 판매수익·페어보너스 자격을 얻는다"는 구조(§3.5.5, D-028)에 대한 다각도 분석:

| 관점 | 분석 |
|---|---|
| 가입 강제 여부 | 회원가입 자체는 무료이고 패키지 구매도 형식상 선택이므로, 가장 직접적인 금지행위 유형("가입조건으로서의 사재기 강요")에는 해당하지 않는 것으로 보인다 — 이미 D-024에서 확인된 부분 |
| 보너스를 매개로 한 구매 유도 | 그러나 "후원수당(보너스) 수령 자격의 조건으로 구매를 요구"하는 패턴은, 가입 강제와는 별개로 다단계판매업자의 금지행위(예: 경제적 이익을 미끼로 한 등록·구매 유도 관련 조항)와 관련될 수 있는 영역이다 — 정확한 조문 적용은 법률 자문 필요 |
| 일반적 "Active Qualification"과의 비교 | 스폰서가 일정 자격(보통 소액의 자기소비)을 유지해야 다운라인 보너스를 받는 구조 자체는 국내외에서 흔히 합법적으로 운영된다. 그러나 FNS의 자격 유지 비용은 §3.5.2의 소액(5만원)이 아니라 **400만원이라는 1회성 고액 구매** — 통상적인 자격유지 비용 수준을 크게 초과해, "구매 강요" 인상을 줄 위험이 일반적인 자격조건보다 크다 |
| 소멸(pass-up 없음) 구조의 함의 | 자격 미충족 시 보너스가 그냥 소멸하므로(누구에게도 이전되지 않음), 추천인이 "내 다운라인이 패키지를 사면 나도 받을 수 있다"는 기대로 적극적으로 다운라인의 패키지 구매를 독려하면서도, 정작 본인이 사지 않으면 그 노력이 전부 무위가 된다 — 결과적으로 추천인 본인이 패키지를 사야 한다는 압박이 모집 활동 과정에서 자연스럽게 발생하는 구조다 |
| 사재기 유도 해당 여부 | "사재기"는 통상 판매하지 못할 재고를 강제로 사들이게 하는 행위를 의미한다. 400만원 패키지가 실제 사용·재판매 가능한 제품 구성인지, 자격 취득을 위한 명목상 구성인지에 따라 평가가 달라진다 — 제품 구성의 실질이 사재기 판단에 영향을 줄 수 있다 |
| 종합 | (i) 가입 자체를 강제하지 않는다는 점에서 가장 직접적인 금지행위는 회피했으나, (ii) 고액(400만원) 구매를 보너스 수령의 전제조건으로 둔 점, (iii) 자격 유지 비용이 일반적 수준(5만원)을 크게 초과하는 점에서 별도의 법적 리스크 패턴에 해당할 가능성을 배제할 수 없다 |

- **권고**: 법무 검토 전까지는 [LEGAL-CHECKLIST.md](LEGAL-CHECKLIST.md) §5의 기존 항목을 유지하고, O-081을 Tier 0급 우선순위로 둔다(이미 §3.1 Tier 0-1 반영됨).

#### 5.10.6 최종 권고안

1. **정산 주기**: 조치 완료(D-029) — 추가 작업 불필요.
2. **직급 체계**: 삭제 권고. 실제 마케팅 플랜에 없고, 구체값이 한 번도 확정된 적이 없으며, 4축 보상엔진과 독립적이다. 다만 PROJECT-CONTEXT.md §1(프로젝트 정의)까지 수정해야 하는 영향력이 큰 변경이므로 **사용자의 명시적 실행 승인 후 별도 라운드에서 진행**을 권고한다.
3. **PV**: 삭제 권고. 이미 자체적으로 "단순화 후보"(O-062 등)로 지목되어 있었고, 모든 실제 계산이 KRW로 동작한다. 직급과 별개로 먼저 제거해도 무방하며, 영향 범위가 직급보다 작다.
4. **수당 명칭("제품 판매수익")**: 한국 MLM 업계 용어 비교 결과는 아래 표 참조. **현재 명칭 유지를 1순위로 권고** — "추천"이라는 단어를 다시 쓰면 D-028에서 분리한 자격조건(본인 구매)의 논리와 충돌하고, O-059 법적 분류 논의를 다시 "후원수당" 쪽으로 강하게 기울게 할 위험이 있다.

   | 후보 | 장점 | 단점 |
   |---|---|---|
   | 추천수당 | 업계에서 친숙, 회원이 바로 이해 | "추천"이 강조되어 O-059가 후원수당 쪽으로 기울고, 본인 구매 자격조건(D-028)과 의미상 충돌("추천만으로 받는 게 아님") |
   | 판매수당 | 간결, 업계에서도 흔히 사용 | "판매"의 주체가 모호함(추천인이 판 게 아니라 다운라인이 산 것) |
   | 소개수당 | 신규회원 최초 실적에 대한 1회성 수당이라는 의미로 업계에서 흔함 | **이미 [COMPENSATION-RULES.md](COMPENSATION-RULES.md) §4 표에 별도의 미채택 후보 수당("소개수당", Referral Bonus)으로 존재** — 이름을 재사용하면 혼란·중복 |
   | 패키지 추천수당 | 의미가 가장 직접적임 | "추천"이 다시 들어가 D-028의 명칭 정정 취지(추천보상→판매수익)와 모순, O-059에 재차 영향 |
   | **제품 판매수익(현재안)** | 본인 구매 자격조건과 가장 일관됨, D-028 논리와 부합 | 한국 MLM 업계에서는 "~수당" 계열 용어가 더 표준적이라 다소 생소할 수 있음 |

   업계 친숙도가 중요하다면 "**패키지 판매수당**"(추천 없이 "판매"+"수당"만 결합)을 절충 대안으로 제시한다 — 익숙한 "수당" 어미를 쓰면서도 "추천"을 빼 법적 뉘앙스를 지킨다.
5. **O-081**: 법무 검토 전까지 현재 보수적 기본값(35% 한도 산정에 포함, [COMPENSATION-RULES.md](COMPENSATION-RULES.md) §6) 유지를 권고. 최종 판단은 보류.
6. ~~**관리자 설정화**: `marketing_plan_versions.plan_definition` 스키마 구체화를 다음 설계 라운드의 최우선 항목으로 권고(5.10.5 스켈레톤 참조)~~ — **완료(D-032, 2026-06-24)**: 스켈레톤을 확정 스키마로 채택했다.

**실행 순서 제안(작성 당시)**: PV 제거(영향 작음) → 직급 체계 제거(영향 큼, 비전 문서까지 수정) → `plan_definition` 스키마 구체화 → 명칭 최종 확정 → O-081 법률 자문 의뢰. **업데이트(2026-06-24)**: 사용자가 직급·PV 제거를 모두 확정해 D-030으로 즉시 실행했다(둘 다 동시에 처리). **추가 업데이트(같은 날, D-031/D-032)**: `plan_definition` 스키마 구체화도 완료했고, D-030 적용 과정에서 남은 잔여 참조 불일치(DO-NOT-TOUCH.md/ARCHITECTURE.md의 폐기 테이블 참조)를 정리했으며, 누락되어 있던 쇼핑몰(정기배송/자동결제) 구조를 신설했다. 남은 항목은 명칭 최종 확정(O-085) → O-081 법률 자문 의뢰 순으로 진행 권고.

### 5.11 FNS 마케팅 플랜 최종 재정의 보고서 — 일반 MLM 구조 배제 점검 (2026-06-24, D-031/D-032)

> 배경: 사용자가 "기존 문서에 일반 MLM 구조(PV, 직급, 주정산 등)가 섞여 들어간 것 같다"는 우려로 FNS 마케팅 플랜을 처음부터 다시 점검할 것을 요청했다. 점검 결과 핵심 구조(자격 분리, 라인당 단일비율, 월정산 단일, 직급/PV 폐기)는 **이미 D-024~D-030으로 반영되어 있었음**을 확인했다 — 본 절은 ① 이미 반영된 내용을 사용자가 요청한 7개 산출물 형식으로 재정리하고, ② 점검 과정에서 새로 발견한 잔여 문제(D-031/D-032)를 함께 보고한다. 코드는 생성하지 않았다.

#### 5.11.1 산출물 1 — FNS 최종 마케팅 플랜 (자격 흐름)

```
무료 회원가입 (구매 비조건, §3.5.1)
   │
   ▼
제품 구매 (월 누적 5만원 이상, 그 달 1일~말일 기준 — §3.5.2)
   │
   ▼
유니레벨 후원수당 자격 (그 달만 유효, 매월 독립 판정 — §3.5.2)
   │                                         ※ 아래 패키지 경로와 완전히 독립(D-028)
   ▼
[선택] 400만원 패키지 구매 (1종, 선택 구매 — §4.1.1)
   │
   ▼
제품 판매수익 자격 (1회 구매로 영구 획득, 유지조건 없음 — §3.5.5)
   │
   ▼
직추천 회원 패키지 구매 → 제품 판매수익 100만원 (§4.1.2)
   │
   ▼
페어보너스 자격 (= 제품 판매수익 자격과 동일 — §3.5.5)
   │
   ▼
직추천 패키지 구매자 2명 페어(30일 윈도우) → 페어보너스 200만원 (§4.1.3)
```

이 흐름은 사용자가 §1에서 제시한 다이어그램과 **일치한다** — [COMPENSATION-RULES.md](COMPENSATION-RULES.md) §3.5/§4.1에 이미 이 구조로 정의되어 있다(D-024/D-028). 유일한 보강 포인트: 사용자 다이어그램은 두 자격이 순차적인 단일 경로처럼 보이지만, 실제 확정 규칙은 **유니레벨 자격(월 단위, 5만원)** 과 **제품판매수익·페어보너스 자격(1회성, 패키지 400만원)** 이 서로 독립이라는 점이다(D-028) — 하나를 충족해도 다른 하나가 자동 충족되지 않는다. 위 다이어그램의 화살표는 "이 순서로 진행할 수 있다"는 일반적 진행 경로를 의미할 뿐, "왼쪽을 충족해야 오른쪽 자격이 생긴다"는 의존관계가 아니다.

#### 5.11.2 산출물 2 — 수당 종류 정의

FNS 마케팅 플랜은 **4개 축**으로 구성된다 ([COMPENSATION-RULES.md](COMPENSATION-RULES.md) §4, D-024):

| 수당 | 트리거 | 산정 기준 | 자격 |
|---|---|---|---|
| 유니레벨 후원수당(조직수당) | 월 배치(월정산) | 라인 도달 깊이별 단일비율(3/3/3/4/5%) × 라인 매출 | §3.5.2 (매월 5만원) |
| 제품 판매수익 | 이벤트(직추천 패키지 구매) | 패키지 가격 × 25%(100만원) | §3.5.5 (본인 패키지 구매 1회) |
| 페어보너스 | 이벤트(페어 성립) | Pair당 200만원 | §3.5.5 (제품 판매수익과 동일) |
| "+알파" 보너스(옵션) | 누적 적립 | 매출 대비 0.1~0.5%/0.2%/0.2%(여행/자동차/자기계발) | 세부 미확정 |

비채택/미확정 수당(소개수당/매칭수당/리더십수당)은 §5.11.5(삭제 가능한 일반 구조) 참조.

#### 5.11.3 산출물 3 — 자격 구조 정의

**2개의 완전히 독립된 자격**이다(D-028, [COMPENSATION-RULES.md](COMPENSATION-RULES.md) §3.5):

| 자격 | 충족 조건 | 판정 주기 | 미충족 시 |
|---|---|---|---|
| ① 유니레벨 후원수당 자격 | 그 달(1일~말일) 누적 제품 구매 5만원 이상 | 매월 독립 판정 | 그 달만 수령 제외, 다음 달 자동 복구 |
| ② 제품 판매수익·페어보너스 자격 | 본인이 400만원 패키지를 구매한 적이 있음(1건 이상) | 1회 충족 시 영구 | 직추천 구매/페어가 성립해도 소멸(pass-up 없음) |

①은 `orders` 합산의 파생값, ②는 `package_purchases` EXISTS 판정의 파생값이다 — 둘 다 별도 컬럼이 아니다([DATABASE.md](DATABASE.md) §3.27).

#### 5.11.5 산출물 5 — 삭제 가능한 MLM 일반 구조 (이미 삭제 완료, 재확인)
